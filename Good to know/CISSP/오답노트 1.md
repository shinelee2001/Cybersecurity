# CISSP 오답노트 1

## 1. 바로 외울 것

### 데이터 보호

* **이동식 매체:** 암호화가 데이터와 함께 이동하므로 분실·탈취 시 가장 직접적인 보호
* **Data at Rest:** 저장 암호화
* **Data in Transit:** TLS, VPN, End-to-End Encryption
* **Data in Use:** IRM/DRM, 사용권한 통제
* **Data Owner:** 분류·보호·접근 요구사항 결정
* **Data Custodian:** 백업·저장·접근통제 등 기술적으로 구현
* **규제 데이터 해외 이전:** 데이터 민감도와 국가별 법률·관할권을 기준으로 통제
* **제3자 규제 데이터 처리:** 규제 준수와 법적 책임을 먼저 확인
* **외부 컨설턴트:** 계약에 기밀유지 및 보안 요구사항 포함

### 네트워크

* **East-West Traffic:** 데이터센터 내부 서버 간 통신
* **North-South Traffic:** 내부와 외부 간 통신
* **Multi-tenant 네트워크 격리:** VRF 또는 별도 Routing Instance
* **DDoS 대응:** CDN, Anycast, Scrubbing 등 분산 방어
* **Satellite 통신:** 광범위하게 수신 가능하므로 강력한 암호화 필요
* **FCoE/iSCSI:** 공유 네트워크에서 블록 데이터가 노출될 수 있으므로 암호화·격리 필요
* **Site-to-Site VPN:** 두 VPN Gateway 사이의 터널 구간만 보호
* **End-to-End Encryption:** 송신 애플리케이션부터 최종 수신자까지 보호

### IAM

* **Federation:** 조직 간 신뢰를 이용해 중복 계정 없이 인증 공유
* **SSO:** 한 번 인증하여 여러 서비스 이용
* **RBAC:** 역할 기반 권한 관리
* **Separation of Duties:** 한 사람이 거래·업무 전 과정을 통제하지 못하도록 분리
* **Least Privilege:** 업무 수행에 필요한 최소 권한만 부여
* **Access Review:** 실무 관리자가 직원의 실제 업무 필요성을 검증
* **Identity Proofing:** 정부 발급 신분증 등 권위 있는 증거 사용
* **KBA:** 개인정보 질문 기반 인증으로 유출 위험 때문에 약한 방식
* **Duress:** 비밀 코드, 강압용 PIN, 위험 문구로 조용히 경보
* **Session Fixation 방지:** 인증 성공 후 Session ID 재생성

### AAA 프로토콜

| 구분    | TACACS+     | RADIUS        |
| ----- | ----------- | ------------- |
| 전송    | TCP 49      | UDP 1812/1813 |
| 암호화   | 전체 Payload  | 비밀번호 필드 중심    |
| AAA   | 인증·권한·감사 분리 | 인증과 권한이 결합    |
| 주요 용도 | 네트워크 장비 관리  | 사용자 네트워크 접근   |

* **Diameter:** RADIUS의 후속 프로토콜이며 TCP/SCTP와 TLS/IPsec 사용 가능

### 보안 모델·설계

* **Bell-LaPadula:** 기밀성 모델

  * No Read Up
  * No Write Down
* **Biba:** 무결성 모델

  * No Read Down
  * No Write Up
* **Secure Defaults:** 명시적으로 허용하지 않으면 비활성화 또는 거부
* **Defense in Depth:** 여러 계층에 중복 통제 배치
* **빠른 케이블:** 성능 개선이지 가용성 통제는 아님
* **Configuration Management:** 변경 권한통제와 변경 기록·감사 추적
* **SBOM:** 릴리스에 포함된 라이브러리와 구성요소 목록

### 운영·사고대응

* **SIEM:** 로그는 입력 데이터이고 탐지 규칙·상관분석이 실제 탐지 엔진
* **Executive IR Report:** 사업 영향, 교훈, 향후 개선 권고
* **기술 세부내용:** 기술 보고서 또는 부록에 배치
* **외부 사고 공지:** 사전 승인된 템플릿을 상황에 맞게 수정
* **CRL 지연:** 폐기된 인증서가 잠시 신뢰될 수 있음
* **Temporary Access:** 제한적이고 단기적인 대체 접근 제공
* **Software Supply Chain:** SBOM, 코드 서명, 구성요소 출처 검증

### 소프트웨어

* **Agile:** 변화하는 요구사항에 반복적·점진적으로 대응
* **DevOps:** 개발·운영 통합과 배포 자동화
* **Secure Code Review:** 실제 코드 취약점 확인
* **Threat Modeling:** 설계 단계에서 위협과 공격경로 식별
* **Open Source 평가:** 유지보수 활동, 커뮤니티, 코드 리뷰, 업데이트 지속성 확인
* **미설정 상태:** Deny/Disable by Default
* **취약점 스캐너:** 최신 Plugin, Signature, Vulnerability Database 중요
* **LLM 개인정보 위험:** 학습 데이터가 모델에 기억되어 출력으로 재노출될 수 있음

## 2. 반드시 구분해서 이해할 것

### 목표와 직접 연결되는 통제 선택

CISSP에서는 단순히 좋은 통제가 아니라 **질문의 목적을 가장 직접적으로 해결하는 통제**를 선택한다.

| 문제의 목적              | 직접적인 통제                 |
| ------------------- | ----------------------- |
| 분실된 USB의 데이터 보호     | 이동식 매체 암호화              |
| 음성 도청 방지            | Media Encryption        |
| 다중 테넌트 네트워크 격리      | VRF                     |
| Session Fixation 방지 | 로그인 후 Session ID 재생성    |
| 대규모 DDoS 흡수         | CDN, Anycast, Scrubbing |
| DB 평문 데이터 보호        | 데이터·필드 암호화              |

보조 통제가 더 익숙하더라도 문제의 목적을 직접 해결하지 않으면 정답이 아니다.

### 기밀성·무결성·가용성

* **Confidentiality:** 비인가 공개 방지
* **Integrity:** 비인가 변경·오염 방지
* **Availability:** 필요할 때 접근 가능
* **Authentication:** 주체의 신원 확인
* **Non-repudiation:** 행위 부인 방지

#### 판별 기준

* 도청·노출 → **Confidentiality**
* 변조·오래된 데이터·순서 오류 → **Integrity**
* 지연·중단·접근 불가 → **Availability**

### Encryption vs Access Control vs DLP

* **Encryption:** 데이터가 유출되어도 내용을 읽지 못하게 함
* **Access Control:** 누가 데이터에 접근할지 제한
* **DLP:** 데이터의 부적절한 전송·반출을 탐지하고 차단

#### 문제 풀이

* 데이터 자체의 기밀성 → **Encryption**
* 누가 데이터를 사용할 수 있는가 → **Access Control**
* 외부 반출·업로드 방지 → **DLP**

### Hash vs MAC vs Digital Signature

| 방식                |    무결성 | 인증 | 부인방지 | 기밀성 |
| ----------------- | -----: | -: | ---: | --: |
| Hash              |      O |  X |    X |   X |
| HMAC/MAC          |      O |  O |    X |   X |
| Digital Signature |      O |  O |    O |   X |
| Encryption        | 경우에 따라 |  X |    X |   O |

* **Hash:** 누구나 다시 계산할 수 있어 송신자를 인증하지 못함
* **HMAC:** 공유키 보유자가 생성했음을 확인하지만 양측이 동일한 키를 가지므로 부인방지 불가
* **Digital Signature:** 개인키로 서명하여 무결성·인증·부인방지 제공

### Data State

* **At Rest:** 파일, DB, Backup 등 저장 상태
* **In Transit:** 네트워크를 통해 이동 중
* **In Use:** 메모리나 애플리케이션에서 처리 중

VPN은 전송 구간만 보호한다. 사용 중인 데이터의 지속적인 사용권한은 IRM/DRM이 담당한다.

### Effectiveness vs Compliance

* **Effectiveness:** 통제가 실제로 위험을 줄이고 정상 작동하는가
* **Compliance:** 법률·규정·표준에서 요구하는 기준을 충족하는가

#### 공급업체 평가

* 서비스 효과성 → 실제 실적, 평판, 성과 증거
* 규제 준수 → 인증, 감사보고서, 규제 통제 충족 여부

인증서는 기준을 충족했다는 증거이지 실제 보안 성능을 완전히 보장하지 않는다.

### Data Owner vs Data Custodian

* **Data Owner**

  * 데이터 분류 결정
  * 접근 요구사항 결정
  * 보존·보호 요구사항 결정
  * 잔여 위험 수용
* **Data Custodian**

  * Owner의 요구사항을 시스템에 구현
  * 백업, 저장, 복구, 접근통제 운영

> Owner decides. Custodian implements.

### SoD vs Least Privilege

* **Separation of Duties:** 업무 단계와 권한을 여러 사람에게 분리
* **Least Privilege:** 개인이 보유한 권한의 범위를 최소화
* **Job Rotation:** 장기간 은폐되는 부정행위 발견
* **Mandatory Vacation:** 다른 사람이 업무를 수행하면서 이상행위 발견

### SSO vs Federation

* **SSO:** 한 번 로그인하여 여러 시스템 이용
* **Federation:** 서로 다른 조직 또는 보안 도메인이 인증 결과를 신뢰

Federation의 핵심 장점은 상대 조직에 중복 사용자 계정을 만들지 않아도 된다는 것이다.

## 3. 평가·감사·침투 테스트

### 독립성

가장 편향이 적고 포괄적인 검증은 **독립된 외부 Penetration Test**이다.

* 내부 팀: 환경 이해도는 높지만 이해관계와 익숙함으로 편향 가능
* 외부 팀: 독립성과 객관성이 상대적으로 높음

### 평가 도구 정확도

좋은 평가 도구는 다음 조건을 만족해야 한다.

* False Positive가 낮음

* False Negative가 낮음

* 최신 취약점 탐지 가능

* 환경에 적합한 정확한 결과 제공

* **False Positive:** 문제가 없는데 문제로 탐지

* **False Negative:** 실제 문제가 있는데 탐지하지 못함

보안에서는 False Negative가 실제 취약점을 놓치기 때문에 특히 위험하다.

### Due Care vs Due Diligence

* **Due Care:** 합리적인 보호조치를 마련하고 수행해야 할 책임
* **Due Diligence:** 보호조치가 실제로 수행되고 효과적인지 지속적으로 확인

소프트웨어 보안을 실제로 확인하려면 다음이 필요하다.

* Secure Code Review
* SAST
* DAST
* Penetration Test
* 취약점 평가
* 독립적 검증

Threat Modeling은 설계 단계의 예방 활동이며 구현된 결과의 안전성을 직접 검증하는 것은 아니다.

### Third-party 평가

* **장기 서비스 신뢰성:** 실적, 평판, 업데이트 역량
* **규제 준수:** 인증, 외부 감사, 계약상 의무
* **민감정보 처리:** 저장 위치, 접근자, 암호화, 삭제, 사고 통지
* **소프트웨어 공급망:** SBOM, 구성요소 출처, 취약점 관리

## 4. 네트워크·통신

### East-West vs North-South

* **East-West:** 내부 서버, VM, Container, Microservice 간 통신
* **North-South:** 사용자·인터넷과 데이터센터 내부 간 통신

현대 데이터센터는 분산 시스템과 마이크로서비스 때문에 East-West 트래픽 비중이 높다. 따라서 경계 방화벽만으로는 부족하고 내부 세분화와 내부 트래픽 모니터링이 필요하다.

### VRF vs VLAN vs Firewall

* **VLAN:** Layer 2 Broadcast Domain 분리
* **VRF:** Layer 3 Routing Table 분리
* **Firewall:** 트래픽 허용·차단 정책 집행

다중 테넌트 간 근본적인 라우팅 격리에는 VRF가 더 직접적이다.

### DDoS

방화벽을 추가하는 것만으로는 대규모 트래픽을 흡수할 수 없다.

효과적인 통제:

* Anycast
* CDN
* Upstream Scrubbing
* Rate Limiting
* 분산 Edge Infrastructure
* ISP와의 협력

### Latency

Latency가 높아 서비스가 느려지거나 접근할 수 없다면 Availability 문제다. 그러나 지연으로 인해 오래된 데이터나 순서가 잘못된 데이터가 사용되면 Integrity 문제가 될 수 있다.

문제에서 제시하는 **최종 결과**를 기준으로 판단한다.

### 통신 암호화

* 음성·영상 도청 방지 → **Media Encryption**
* 세션 설정 메시지 보호 → **Signaling Protection**
* Satellite 통신 → 광범위한 전파 특성 때문에 강력한 암호화
* Site-to-Site VPN → Gateway 간 구간 보호
* 민감 데이터 자체 보호 → Payload 또는 End-to-End Encryption

## 5. 사고대응·운영

### 외부 커뮤니케이션

사고 발생 후 발표문을 처음부터 만들면 오류와 법적 위험이 커진다.

사전에 준비할 것:

* 승인된 공지 템플릿
* 공식 대변인
* 법무 검토 절차
* 규제기관 통지 절차
* 고객·언론·파트너별 메시지 체계

### Executive IR Report

경영진은 명령어와 로그보다 다음 정보를 필요로 한다.

* 사업 영향
* 법률·규제 위험
* 재무 영향
* 사고 원인
* Lessons Learned
* 개선 권고
* 책임자와 일정
* 필요한 예산

### SIEM 효과 향상

로그 수가 많다고 탐지 성능이 자동으로 좋아지는 것은 아니다.

1. 핵심 Use Case 정의
2. 필요한 로그 수집
3. 최신 위협정보 반영
4. Detection Rule 튜닝
5. False Positive 감소
6. 대응 절차와 연계
7. 탐지 성능 측정

> Logs are the fuel. Detection logic is the engine.

### CRL

인증서가 폐기되어도 CRL 배포와 조회가 지연되면 기존 인증서가 잠시 신뢰될 수 있다.

* **CRL:** 폐기 인증서 목록을 주기적으로 배포
* **OCSP:** 특정 인증서 상태를 비교적 실시간으로 조회

### Temporary Access

스마트카드·토큰 분실 시 영구적인 우회 계정을 발급하면 안 된다.

안전한 임시 접근:

* 사용자 신원 재확인
* 관리자 승인
* 짧은 유효시간
* 제한된 권한
* 활동 기록
* 사용 후 자동 폐기

## 6. 소프트웨어 보안

### Agile vs DevOps

* **Agile:** 요구사항 변화에 대응하는 개발 방법론
* **DevOps:** 개발·운영 협업과 CI/CD 자동화

#### 문제 풀이

* 요구사항이 자주 변경됨 → **Agile**
* 배포 속도와 자동화 필요 → **DevOps**

### Threat Modeling vs Code Review vs Penetration Test

* **Threat Modeling:** 설계가 어떤 공격을 받을 수 있는지 예측
* **Code Review:** 구현된 코드에서 취약점을 확인
* **Penetration Test:** 실행 중인 시스템을 실제로 공격하여 악용 가능성 검증

### Open Source 평가

오픈소스라고 자동으로 안전하거나 위험한 것은 아니다.

확인할 것:

* 최근 Commit과 Release
* 취약점 대응 속도
* 활성 Maintainer 수
* 커뮤니티 활동
* 코드 리뷰 절차
* 장기 지원 가능성
* 의존성 및 라이선스

### Secure Defaults

사용자가 별도로 설정하지 않아도 안전해야 한다.

* 불필요 기능 비활성화
* 기본 접근 거부
* 최소 권한
* 강한 초기 설정
* 오류 발생 시 Fail Secure

### Session Fixation

공격자가 미리 생성한 Session ID를 피해자에게 사용하게 하고, 피해자가 로그인한 뒤 동일한 세션을 탈취하는 공격이다.

대응:

* 로그인 성공 후 Session ID 재생성
* 권한 변경 후 Session ID 재생성
* 로그아웃 시 세션 폐기
* Secure Cookie
* HttpOnly Cookie
* SameSite Cookie
* 짧은 Session Timeout

### SBOM

SBOM은 소프트웨어에 포함된 다음 정보를 제공한다.

* 오픈소스 라이브러리
* 버전
* 공급업체
* 의존성
* 구성요소 관계

새로운 취약점이 발표되면 영향을 받는 제품과 버전을 빠르게 찾는 데 사용한다.

### LLM 개인정보

민감정보를 포함해 학습된 모델은 다음 위험이 있다.

* 학습 데이터 일부 암기
* Prompt를 통한 개인정보 재출력
* 출력 또는 추론 결과에 민감정보 포함
* 삭제 요청 후에도 모델에 영향 잔존

주요 통제:

* 데이터 최소화
* 비식별화·가명화
* 민감정보 학습 제외
* 출력 필터링
* 접근통제
* 학습 데이터 출처 관리
* 모델 및 데이터 삭제 절차

## 7. 인식 교육·거버넌스

### Security Awareness

최종 목적은 교육 이수율이 아니라 **실제 보안 행동 변화**다.

좋은 교육 콘텐츠:

* 피싱
* 비밀번호와 MFA
* 데이터 처리
* 이동식 매체
* 물리 보안
* 사고 신고
* 실제 업무 시나리오

너무 기술적인 공격 설명보다 사용자가 무엇을 해야 하며 왜 해야 하는지를 알려주는 것이 중요하다.

### 비례성

모든 데이터를 무조건 암호화하거나 모든 국외 이전을 차단하는 것은 비즈니스 요구를 과도하게 제한할 수 있다.

CISSP가 선호하는 접근:

1. 데이터 민감도 확인
2. 법률·규제 요구사항 확인
3. 사업 필요성 확인
4. 위험 평가
5. 위험에 비례하는 통제 적용

---

## 8. 문제 풀이 기준

### 주요 키워드

* **PRIMARY:** 근본 목적 또는 핵심 기능
* **MOST EFFECTIVE:** 위험을 가장 직접적으로 줄이는 통제
* **BEST:** 사업·위험·비용을 종합한 최선의 선택
* **LEAST:** 목적과 가장 관련이 적은 보기
* **EXECUTIVE:** 기술보다 사업 영향과 개선 방향
* **TRANSITION:** 기존 시스템과의 통합·호환성
* **IN USE / AT REST / IN TRANSIT:** 데이터 상태
* **COMPLIANCE:** 규제·표준 충족
* **EFFECTIVENESS:** 실제 통제 성능

### 선택 순서

1. 보호하려는 자산 확인
2. 위협과 요구되는 CIA 속성 확인
3. 관리적·기술적·물리적 통제 수준 확인
4. 목표를 가장 직접적으로 해결하는 통제 선택
5. 지나치게 구체적이거나 과도한 보기 제거
6. 법률·안전·비즈니스가 기술보다 우선인지 확인

## 9. 문항 해설을 그대로 외우면 안 되는 항목

### 음성·영상 협업의 RBAC

문항 정답은 RBAC였지만 실제 환경에서 기밀성이 핵심이면 End-to-End Encryption이 더 중요할 수 있다.

* 기능별 접근권한과 사용 편의성 → RBAC
* 통화 내용 도청 방지 → End-to-End Encryption

### Latency와 Integrity

느린 네트워크 자체는 Availability 문제다. 다만 지연으로 오래된 데이터나 순서가 잘못된 데이터가 처리되면 Integrity 문제로 해석될 수 있다.

### 개방형 Wi-Fi

문항은 강한 무선 암호화를 정답으로 보았지만, 승인되지 않은 무선망이거나 즉시 안전하게 설정할 수 없다면 실제 대응은 격리·차단이 우선일 수 있다.

### Secure Coding Training

개발자 교육은 예방 통제지만 소프트웨어가 실제로 안전하다는 증거는 아니다.

실질적인 Due Diligence에는 다음이 필요하다.

* Secure SDLC
* Code Review
* SAST/DAST
* Penetration Test
* 취약점 관리
