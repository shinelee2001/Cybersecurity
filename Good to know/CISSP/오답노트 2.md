# CISSP 오답노트 2

## 1. 바로 외울 것

### 네트워크

* **VDOM:** 한 물리 장비에서 여러 독립된 논리 장비·방화벽 인스턴스를 운영
* **VDOM 목적:** 부서·고객·테넌트 간 논리적 네트워크 격리
* **Proxy Server:** 클라이언트를 대신해 외부 서비스에 요청하고 내부 주소·신원을 은닉
* **WAF:** HTTP/HTTPS 기반 웹 공격 탐지·차단
* **Zigbee:** IoT용 저전력·저속·메시 네트워크
* **TCP/IP 4계층:** Link → Internet → Transport → Application
* **MAC 주소 변경 도구:** MAC Filtering 또는 Port Security 우회에 사용 가능

### 침투 테스트

* **Rules of Engagement:** 침투 테스트의 허용 범위, 금지 행위, 시간, 대상, 중단 조건 정의
* **Statement of Work:** 업무 범위, 일정, 비용, 산출물 정의
* **Service Level Agreement:** 서비스 성능과 가용성 기준 정의
* **Integration Test:** 새 시스템과 기존 시스템 간 인터페이스·데이터 교환 검사
* **Regression Test:** 변경 후 기존 기능이 정상적으로 유지되는지 검사

### 접근통제·보안 아키텍처

* **RBAC의 핵심 장점:** 역할 단위 Provisioning·Deprovisioning으로 관리 효율 향상
* **Reference Monitor:** Subject와 Object 사이의 모든 접근을 중재·검사·집행
* **Trusted Computing Base:** 보안정책을 집행하는 전체 하드웨어·소프트웨어 집합
* **HSM:** 암호키를 변조 방지 하드웨어 내부에서 생성·보관·사용
* **Secure Boot:** 부팅 구성요소의 디지털 서명을 검증
* **Measured Boot:** 부팅 구성요소의 측정값을 TPM 등에 기록
* **SAML:** 조직·도메인 간 인증 Assertion을 전달하는 Federation 표준
* **IAL2:** 신원 증거를 제출하고 원격 또는 대면 검증을 수행한 수준

### 암호화

* **Hash:** 무결성 제공, 송신자 인증 불가
* **Digital Signature:** 무결성 + 인증 + 부인방지
* **AES:** 기밀성 제공
* **HSM:** 암호화 알고리즘 자체보다 암호키 보호에 초점
* **Key Distribution:** 조직의 Cryptographic Key Management Policy에 따라 수행
* **Field-level Encryption:** DB 내부의 특정 민감 필드를 직접 암호화
* **Full-disk Encryption:** 디스크 분실·오프라인 탈취 방어

### 운영·복구

* **MTTF:** 수리하지 않는 장비가 처음 고장 날 때까지의 평균시간
* **MTBF:** 수리 가능한 장비의 고장과 다음 고장 사이 평균 가동시간
* **MTTR:** 고장 후 평균 수리·복구시간
* **Malware 제거 실패:** Known-good Baseline 또는 신뢰 가능한 백업으로 재구축
* **Data Remanence:** 삭제 후에도 저장매체에 남아 복구 가능한 잔존 데이터
* **Patch 우선순위:** 취약점 위험도와 자산 중요성이 Maintenance Window보다 우선

### 물리·환경 보안

* **서버실 화재:** Clean-agent 또는 Gas-based Fire Suppression
* **과열:** 온도·습도 센서 및 HVAC 모니터링
* **물 스프링클러:** 인명 보호에는 유효하지만 전자장비 손상 가능
* **EPO:** 비상 시 전원을 차단하지만 화재·과열을 직접 탐지하지는 않음

## 2. 반드시 구분해서 이해할 것

### Risk Appetite vs Risk Tolerance

* **Risk Appetite:** 조직이 목표 달성을 위해 전체적으로 감수할 의사가 있는 위험 수준
* **Risk Tolerance:** 특정 업무·시스템에서 허용 가능한 위험의 편차와 한도

#### 예시

* “우리 회사는 신기술 사업에서 높은 위험을 감수한다.” → **Risk Appetite**
* “결제 시스템은 연간 최대 30분의 중단만 허용한다.” → **Risk Tolerance**

### Risk Response

* **Mitigation:** 통제를 적용하여 위험의 가능성 또는 영향을 감소
* **Acceptance:** 위험을 알고도 별도 조치 없이 수용
* **Avoidance:** 위험을 발생시키는 활동 자체를 중단
* **Transfer:** 보험·계약·외주 등을 통해 재정적 책임 일부 이전
* **Residual Risk:** 통제를 적용한 후에도 남아 있는 위험

#### 문제 판별

* 통제를 적용해 위험을 낮춤 → **Mitigation**
* 시간·예산을 고려해 어느 수준까지 감수할지 결정 → **Risk Tolerance**
* 조치하지 않기로 공식 승인 → **Acceptance**
* 위험 활동 자체를 제거 → **Avoidance**

### Safeguard 비용

일반적으로 통제 비용은 예상되는 손실 감소 효과보다 과도해서는 안 된다.

```text
Safeguard Cost ≤ Expected Risk Reduction
```

다만 생명, 법적 의무, 규제 위반처럼 비용만으로 판단할 수 없는 위험도 있다.

### Patch 우선순위

패치의 긴급성은 다음 요소로 결정한다.

1. 취약점 심각도
2. 실제 악용 여부
3. 자산 중요도
4. 인터넷 노출 여부
5. 보상 통제 존재 여부
6. 패치 적용에 따른 운영 위험

**Critical RCE가 실제 악용 중이라면 정기 Maintenance Window를 기다리지 않고 긴급 변경 절차를 사용한다.**

### 암호키 정책 vs 암호기술

Diffie-Hellman이나 HSM은 키 관리의 구현 기술이다. 키 생명주기 전체는 조직 정책이 지배한다.

키 관리 정책에 포함될 항목:

* 키 생성
* 배포
* 저장
* 사용
* 회전
* 백업·복구
* 폐기
* 책임분리
* 키 유출 대응

> 기술을 먼저 선택하는 것이 아니라 승인된 정책에 맞는 기술을 사용한다.

## 3. XSS vs CSRF

### XSS

공격자의 스크립트가 피해자가 접속한 신뢰 사이트의 컨텍스트에서 실행된다.

* 공격 대상: 사용자
* 악용되는 신뢰: 사용자가 웹사이트를 신뢰
* 주요 결과:

  * 세션 탈취
  * 악성 스크립트 실행
  * 화면 변조
  * 사용자 정보 유출

### CSRF

로그인된 사용자의 브라우저와 세션을 이용해 사용자가 원하지 않은 요청을 전송한다.

* 공격 대상: 서버의 상태 변경 기능
* 악용되는 신뢰: 웹사이트가 인증된 사용자를 신뢰
* 주요 결과:

  * 비밀번호 변경
  * 송금
  * 이메일 변경
  * 권한 변경

### 암기

```text
XSS = 신뢰 사이트에서 Script 실행
CSRF = 로그인 세션으로 Forged Request 실행
```

### 주요 대응

#### XSS 대응

* Output Encoding
* Input Validation
* Content Security Policy
* HttpOnly Cookie
* 안전한 DOM API 사용

#### CSRF 대응

* Anti-CSRF Token
* SameSite Cookie
* 중요 작업 재인증
* Origin·Referer 검증

## 4. Hash vs Digital Signature vs Encryption

| 방식                | 기밀성 | 무결성 | 인증 | 부인방지 |
| ----------------- | --: | --: | -: | ---: |
| Hash              |   X |   O |  X |    X |
| HMAC              |   X |   O |  O |    X |
| Digital Signature |   X |   O |  O |    O |
| Encryption        |   O | 제한적 |  X |    X |

### Hash

메시지가 변경되었는지는 확인할 수 있지만 누구나 Hash를 다시 계산할 수 있으므로 송신자를 인증하지 못한다.

### HMAC

공유 비밀키를 사용하므로 무결성과 공유키 보유자 인증을 제공한다. 양측이 같은 키를 보유하므로 부인방지는 제공하지 못한다.

### Digital Signature

개인키로 서명하고 공개키로 검증한다.

제공 기능:

* 무결성
* 송신자 인증
* 부인방지

### Encryption

데이터를 읽지 못하게 하여 기밀성을 제공한다. 암호화만으로 송신자의 신원을 증명하지는 못한다.

## 5. Full-disk vs Field-level Encryption

### Full-disk Encryption

디스크 전체를 암호화한다.

효과적인 상황:

* 노트북·서버 디스크 분실
* 저장장치 물리적 탈취
* 폐기된 디스크의 오프라인 분석

한계:

* 시스템이 부팅되고 디스크가 마운트된 상태에서는 데이터가 복호화됨
* 인증된 DB 사용자 또는 탈취된 계정의 접근은 막지 못할 수 있음

### Field-level Encryption

DB의 특정 열이나 민감 필드를 애플리케이션 또는 DB 계층에서 암호화한다.

효과적인 상황:

* SSN
* 카드번호
* 의료정보
* 비밀번호 이외의 민감 개인정보
* DB Dump 유출

문제에서 **“DB에 민감정보가 평문으로 저장되어 있다”**고 하면 Field-level Encryption이 가장 직접적인 해결책이다.

## 6. 데이터 역할

### Data Owner

* 데이터 분류 결정
* 접근 요구사항 승인
* 보존·보호 요구사항 결정
* 위험 수용
* 사업적 책임 보유

### Data Steward

* 데이터 정의와 분류 기준 관리
* 데이터 품질 관리
* 메타데이터 관리
* 처리 규칙과 정책 준수 관리
* 조직 내 데이터 거버넌스 실무 수행

### Data Custodian

* 저장소 운영
* 백업·복구
* 기술적 접근통제 구현
* 데이터 보존·폐기 조치 수행

### Data Processor

* Data Controller의 지시에 따라 개인정보를 처리

### 암기

```text
Owner decides
Steward governs
Custodian implements
Processor processes
```

## 7. BC/DR 테스트

| 테스트                    | 특징                        |
| ---------------------- | ------------------------- |
| Checklist              | 담당자가 계획 문서를 개별 검토         |
| Tabletop               | 회의실에서 시나리오와 대응 절차 토론      |
| Structured Walkthrough | 담당자들이 계획의 각 단계를 함께 검토     |
| Simulation             | 가상 재난 상황을 제시하고 대응 절차 연습   |
| Parallel               | 복구 사이트를 가동하되 주 시스템도 계속 운영 |
| Functional Drill       | 전원·연락망 등 특정 기능만 시험        |
| Full-scale             | 계획 전체를 실제 상황처럼 전면 가동      |
| Interruption           | 실제 운영을 중단하고 복구 사이트로 전환    |

### 문제 판별

* 문서만 검토 → **Checklist**
* 회의실에서 대응 절차 토론 → **Tabletop**
* 가상의 정전·화재 상황에 대응 → **Simulation**
* 복구 사이트를 함께 가동하지만 운영 전환은 하지 않음 → **Parallel**
* 데이터센터 전체 장애와 계획 전면 활성화 → **Full-scale**
* 실제 서비스를 중단하고 전환 → **Interruption**

> 모의시험에서는 Full-scale과 Interruption의 표현이 혼용될 수 있으므로 실제 운영 중단 여부를 확인한다.

## 8. Integration vs Regression Test

### Unit Test

개별 함수·클래스·모듈이 독립적으로 정상 작동하는지 확인한다.

### Integration Test

여러 시스템·모듈 사이의 인터페이스와 데이터 교환을 확인한다.

확인 대상:

* API
* Legacy Application 연동
* 인증 연계
* 데이터 형식
* 네트워크 통신
* 오류 처리

### Regression Test

변경 후 기존 기능과 보안 기능이 깨지지 않았는지 확인한다.

### Acceptance Test

시스템이 사용자·사업·계약 요구사항을 충족하는지 확인한다.

### 문제 판별

```text
새 시스템과 Legacy 시스템의 상호작용 = Integration
업데이트 후 기존 기능 유지 = Regression
개별 모듈 검사 = Unit
사업 요구 충족 여부 = Acceptance
```

## 9. Reference Monitor vs TCB

### Reference Monitor

Subject가 Object에 접근할 때마다 보안정책에 따라 허용 여부를 판단하는 추상적 개념이다.

필수 속성:

* **Complete Mediation:** 모든 접근을 중재
* **Tamperproof:** 변조할 수 없어야 함
* **Verifiable:** 충분히 작고 단순하여 검증 가능해야 함

### Security Kernel

Reference Monitor 개념을 시스템에 실제로 구현한 핵심 부분이다.

### Trusted Computing Base

시스템 보안정책을 집행하는 모든 보호 메커니즘의 집합이다.

포함 가능 요소:

* 하드웨어
* 운영체제 Kernel
* 보안 모듈
* 접근통제 구성요소
* Firmware

### 암기

```text
Reference Monitor = 접근을 중재하는 개념
Security Kernel = Reference Monitor의 구현
TCB = 보안정책을 집행하는 전체 집합
```

## 10. RBAC의 핵심

RBAC는 사용자에게 권한을 직접 하나씩 부여하지 않고 역할에 권한을 연결한다.

```text
User → Role → Permission
```

주요 장점:

* Provisioning 자동화
* Deprovisioning 단순화
* 관리 오버헤드 감소
* 권한 일관성 향상
* 직무 변경 대응 용이
* Access Review 단순화

RBAC가 Non-discretionary Access를 제공할 수 있지만, 시험에서 **가장 큰 실무적 장점**은 역할 단위 권한 관리의 효율성이다.

## 11. Proxy vs WAF vs Load Balancer

### Proxy Server

클라이언트를 대신해 외부 서비스와 통신한다.

기능:

* 내부 주소 은닉
* 요청 중계
* 콘텐츠 필터링
* 캐싱
* 접근정책 적용

### Reverse Proxy

외부 사용자의 요청을 대신 받아 내부 서버로 전달한다.

기능:

* 내부 서버 은닉
* TLS 종료
* 요청 필터링
* 부하분산
* 캐싱

### WAF

HTTP 요청의 웹 공격 패턴을 분석한다.

방어 대상:

* SQL Injection
* XSS
* Path Traversal
* 악성 HTTP 요청
* 일부 Application-layer DoS

### Load Balancer

요청을 여러 서버에 분산해 성능과 가용성을 높인다.

### 문제 판별

* 클라이언트 대신 외부에 요청 → **Forward Proxy**
* 내부 웹서버 대신 외부 요청 수신 → **Reverse Proxy**
* 웹 공격 차단 → **WAF**
* 여러 서버에 트래픽 분배 → **Load Balancer**

## 12. VDOM

VDOM은 한 물리 네트워크 장비를 여러 독립된 논리 장비처럼 분리한다.

각 VDOM은 별도로 가질 수 있다.

* Routing Table
* Firewall Policy
* Administrator
* Interface
* VPN
* Security Policy

### 목적

* 부서 간 격리
* 고객·테넌트 간 격리
* 관리 영역 분리
* 하드웨어 비용 절감
* 정책 충돌 방지

VDOM은 물리적 분리가 아니라 **논리적 분리**다.

## 13. 무선 기술

### Zigbee

* IEEE 802.15.4 기반
* 저전력
* 저속
* 짧은 거리
* 메시 네트워크 지원
* 센서·스마트홈·IoT에 적합

### Bluetooth

* 개인 장치 간 단거리 통신
* Zigbee보다 일반적으로 높은 데이터 전송률
* 헤드셋·키보드·스마트폰 연결에 주로 사용

### RFID

* 태그 식별과 추적
* 재고·출입카드·물류에 사용
* 일반적인 IoT 메시 통신 프로토콜은 아님

### Li-Fi

* 가시광선을 이용한 데이터 통신
* 빛이 닿는 영역에서 사용
* 일반 IoT 환경에서는 Zigbee보다 덜 일반적

## 15. Software Assurance Level

Software Assurance Level은 애플리케이션 위험과 중요도에 맞춰 필요한 검증 강도를 결정한다.

높은 보증 수준에서 요구될 수 있는 활동:

* 엄격한 보안 요구사항
* Threat Modeling
* 정적·동적 분석
* 코드 리뷰
* 침투 테스트
* 공급업체 증거
* 독립적 평가

핵심 목적은 Compliance 점수 자체가 아니라 **고객의 위험과 요구에 맞는 신뢰 수준을 정하는 것**이다.

## 16. Security Awareness

Security Awareness의 최종 목적은 교육 수료가 아니라 **행동 변화**다.

효과적인 교육의 특징:

* 경영진 지원
* 역할별 교육
* 실제 사례
* 피싱 시뮬레이션
* 반복 교육
* 측정 가능한 결과
* 즉시 적용 가능한 행동 지침

평가 지표 예시:

* 피싱 클릭률
* 신고율
* 교육 후 행동 변화
* 정책 위반 감소
* 사고 신고시간 단축

## 17. NIST CSF Identify

Identify 기능은 조직이 무엇을 보호해야 하는지 파악하는 단계다.

주요 활동:

* 자산 식별
* 중요 업무 식별
* 시스템·데이터 의존성 파악
* 공급업체 의존성 파악
* 위험평가
* 거버넌스 구조 확인

예시:

```text
중요 업무 프로세스
→ 지원 서버
→ 데이터
→ 네트워크
→ 외부 공급업체
```

Awareness Training은 Identify보다 Protect 기능에 더 가깝다.

## 19. 문항 해설을 그대로 외우면 안 되는 항목

### Criminal History와 PII

문항은 범죄기록을 공개정보라서 PII가 아니라고 처리했지만 지나치게 단정적인 설명이다.

공개된 정보라도 특정 개인과 연결되면 개인정보가 될 수 있으며, 범죄기록은 관할권에 따라 민감정보로 강하게 보호될 수 있다.

이 문항의 답은 보기 구성상 정답으로만 기억하고 일반 원칙으로 확대하지 않는다.

### Availability 문항

다음 두 표현은 모두 Availability 개념에 해당할 수 있다.

* 시스템 구성요소가 필요할 때 사용 가능
* 인가된 주체가 필요할 때 객체에 접근 가능

파일 접근 상황에서는 두 번째 표현이 더 구체적이지만 첫 번째 표현도 개념적으로 틀렸다고 보기는 어렵다.

### Third-party CRM 문항

문항에서는 Frequent Update를 정답으로 제시했지만, 민감 고객정보와 통합하기 전이라면 Data Flow Analysis도 매우 중요한 평가다.

* **통합 전 노출 지점 확인:** Data Flow Analysis
* **장기적 취약점 관리:** Vendor Update Capability

문항의 정답 품질이 낮으므로 문맥에 따라 판단한다.

### Secure Boot 문항

* **Secure Boot:** 부팅 구성요소 서명 검증
* **Measured Boot:** 부팅 상태 측정·기록
* **Application Allowlisting:** 승인된 애플리케이션만 실행

일반 애플리케이션 실행까지 제한하는 목적이라면 Application Allowlisting이 더 정확하다.

### Gas-based Fire Suppression

Clean-agent 소화설비는 전자장비 손상을 줄이지만 실제 설계에서는 다음도 고려해야 한다.

* 인명 안전
* 밀폐 공간
* 산소 농도
* 환경 규제
* 경보·대피 절차
* 지역 소방 법규

## 20. 문제 풀이 기준

### 핵심 키워드

* **PRIMARY:** 가장 근본적인 목적
* **MOST IMPORTANT:** 다른 판단을 지배하는 최우선 요소
* **MOST EFFECTIVE:** 위험을 가장 직접적으로 줄이는 통제
* **BEST:** 사업·위험·비용을 종합한 최선
* **NEXT:** 현재 단계 직후의 절차
* **BEFORE:** 선행되어야 하는 정책·승인·절차
* **WOULD NOT:** 포함되지 않는 보기 선택

### 선택 순서

1. 질문이 요구하는 정확한 목적 확인
2. 자산과 위협 확인
3. 요구되는 CIA 속성 확인
4. 정책·절차와 기술 중 어느 수준의 답인지 판단
5. 문제를 직접 해결하는 통제 선택
6. 지나치게 구체적이거나 부차적인 보기 제거
7. 실제 업무와 모의시험 정답이 충돌하면 문항의 표현을 재확인
