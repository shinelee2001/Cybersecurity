# CISSP 오답노트 4

## 1. 바로 외울 것

### 네트워크

* **가장 세밀한 트래픽 통제:** Firewall Rule
* **VLAN:** Layer 2 Broadcast Domain 분리
* **Subnet:** IP 주소·Routing 영역 분리
* **Router ACL:** 주로 L3/L4 기반 Stateless Filtering
* **Firewall:** Source·Destination·Port·Protocol·State·Application 기반 통제
* **IDS 전체 네트워크 가시성:** Network Fabric 전반에 Sensor 배치
* **SDN 핵심 위험:** 중앙 Controller가 Single High-value Target이 됨
* **IPv6:** Broadcast를 사용하지 않고 Multicast로 대체
* **VPN 성능 때문에 우회 발생:** 사용자를 처벌하기 전에 VPN 성능 문제 해결
* **Egress Monitoring:** 정상 Outbound Traffic을 Baseline한 뒤 이상 편차 탐지

### 모니터링·평가

* **Log Review 핵심:** 적시에 정기적으로 검토
* **Log Collection 설계:** 필요한 Event Source와 Event Type이 수집되는지 확인
* **Log Retention:** 사후 조사·규제 준수를 위한 보존
* **Control Coverage Analysis:** 식별된 각 위험에 충분한 통제가 매핑됐는지 확인
* **Continuous Monitoring:** Baseline → Deviation Detection → Alert → Investigation
* **Threat Intelligence Feed:** 다양성뿐 아니라 관련성·정확성·적시성·실행 가능성 확인

### IAM

* **계획된 일시적 관리자 작업:** JIT Privilege + PAM
* **Break-glass Account:** 긴급 상황에서만 사용
* **Dedicated Admin Account:** 일반 계정과 관리자 계정을 분리하지만 Standing Privilege는 남음
* **Access Recertification:** 기존 권한이 여전히 업무상 필요한지 주기적으로 재검증
* **클라우드 Custodian 외주:** 운영은 위임 가능하지만 Data Owner의 책임과 감독은 내부에 유지
* **Server Login 보호:** MFA
* **Fail Secure:** 장애 시 허용이 아니라 안전한 거부 상태로 전환

### 클라우드·공급업체

* **Cloud Internal Audit:** 고객이 통제하는 Configuration·IAM·Policy 준수에 집중
* **Critical Vendor:** 중단되면 핵심 업무가 멈추는 공급업체
* **High-risk Vendor:** 보안통제 부족 등으로 침해 가능성·노출이 큰 공급업체
* **FedRAMP:** 미국 연방정부용 Cloud Security Assessment·Authorization Program
* **SBOM:** 소프트웨어에 포함된 Third-party·Open-source 구성요소 목록
* **Vendor Custodianship:** Custody는 외주 가능하지만 Accountability는 외주 불가

### 소프트웨어·SDLC

* **Waterfall에서 Design Flaw 발견:** Change Request를 통해 Design 단계로 돌아가 수정
* **Agile Security:** 팀 내부에 Security Champion 배치
* **Cleartext Password:** 절대 저장 금지
* **Password Storage:** Salted Adaptive Hash 사용
* **Coordinated Disclosure:** 신속히 접수 확인 후 연구자와 합리적인 공개 일정 협의
* **취약한 Third-party Component:** SBOM으로 구성요소와 영향 범위 식별
* **Fail-secure Design:** 오류·장애 발생 시 Default Deny
* **Secure Defaults:** 초기 설정이 안전한 상태
* **Defense in Depth:** 여러 독립 통제를 계층적으로 배치

### 위험·패치·사고대응

* **Patch 우선순위:** 취약점 심각도·악용 가능성·자산 중요도·노출도를 종합
* **미적용 Patch 발견:** 이미 악용됐는지 먼저 확인
* **Patch Process 개선:** 사고 범위 확인과 긴급 대응 후 수행
* **암호 알고리즘 폐기:** 영향과 의존성 평가 후 데이터 Migration·Re-encryption
* **내부 직원의 자원 오용:** Administrative Investigation부터 시작
* **Global Data Handling:** 적용 법률을 파악하고 가장 엄격한 공통 기준을 Baseline으로 고려

### 물리·전력

* **순간 정전 무중단:** UPS
* **장시간 정전:** Generator
* **Server Room:** 공용 공간과 분리된 Secure Perimeter 필요
* **물리적 출입통제:** Card Access·Mantrap·CCTV·Visitor Control 등을 위험에 맞게 조합
* **Server Room 위치 자체가 부적절:** 출입인증 추가보다 안전한 독립 구역으로 이전

## 2. Network Segmentation 통제 수준

| 통제                | 주요 역할                          |   세분성 |
| ----------------- | ------------------------------ | ----: |
| Subnet            | IP 주소·Routing 영역 분리            |    낮음 |
| VLAN              | Layer 2 Broadcast Domain 분리    |    낮음 |
| Router ACL        | IP·Port·Protocol 기반 필터링        |    중간 |
| Stateful Firewall | Session State 기반 통제            |    높음 |
| NGFW              | Application·User·Content 기반 통제 | 매우 높음 |

VLAN과 Subnet은 **경계를 만든다.** 실제로 어떤 통신을 허용할지는 ACL이나 Firewall이 결정한다.

```text
VLAN/Subnet = 구역 분리
ACL = 패킷 조건 필터링
Firewall = 상태·흐름·애플리케이션 기반 정책 집행
```

문제에서 **MOST granular control over traffic**을 묻는다면 Firewall Rule이 가장 직접적이다.

## 3. Log Collection vs Log Review

### Log Collection 설계

* 어떤 시스템에서 로그를 받을 것인가
* 어떤 Event Type을 기록할 것인가
* Timestamp가 동기화되어 있는가
* 로그가 변조되지 않는가
* 필요한 기간 동안 보존되는가

### Log Review 운영

* 누가 검토하는가
* 얼마나 자주 검토하는가
* 얼마나 빨리 Alert를 조사하는가
* Escalation 기준은 무엇인가
* 검토 결과를 추적하는가

```text
Collection = 무엇을 기록할 것인가
Review = 기록된 것을 언제 확인할 것인가
Retention = 얼마나 오래 보관할 것인가
```

질문이 **“conducting log reviews”**라면 검토 주기와 적시성이 핵심이다. 다만 필요한 Event가 기록되지 않으면 아무리 자주 검토해도 의미가 없으므로 전체 Logging Program에서는 Coverage도 필수다.

## 4. Control Coverage Analysis

Control Coverage Analysis는 식별된 위험과 통제를 매핑하여 미통제 영역을 찾는다.

| Risk              | Control                 | Coverage |
| ----------------- | ----------------------- | -------- |
| Credential Theft  | MFA                     | Covered  |
| Data Exfiltration | DLP·Egress Monitoring   | Partial  |
| Ransomware        | EDR·Backup·Segmentation | Covered  |
| Vendor Failure    | BCP·Alternate Supplier  | Missing  |

```text
Risk Identification
→ Control Mapping
→ Coverage Gap 확인
→ 추가 통제 설계
→ Residual Risk 평가
```

* **Log Review:** 발생한 Event 분석
* **Synthetic Transaction:** 가용성·성능 확인
* **Integration Test:** Component 간 연동 확인
* **Coverage Analysis:** 위험이 통제로 충분히 다뤄지는지 확인

## 5. 계획된 관리자 작업: JIT vs Admin Account

### JIT Privileged Access

```text
작업 요청
→ 승인
→ 필요한 권한만 일시 부여
→ 작업 수행·기록
→ 자동 회수
```

장점:

* Standing Privilege 제거
* 권한 노출시간 감소
* 승인 근거 기록
* Session Monitoring 가능
* 작업 종료 후 자동 회수

### Dedicated Admin Account

일반 업무 계정과 관리자 계정을 분리하여 공격면을 줄이지만 관리자 권한이 계속 존재할 수 있다.

### Break-glass Account

* 긴급 장애
* 정상 인증체계 실패
* 즉각적인 복구 필요

일상적인 유지보수에 Break-glass Account를 사용하면 안 된다.

```text
Routine Maintenance = JIT/PAM
Emergency = Break-glass
Ongoing Admin Duty = Separate Named Admin Account
```

## 7. Critical Vendor vs High-risk Vendor

### Criticality

공급업체가 실패하거나 중단됐을 때 조직에 발생하는 **Business Impact**다.

판단 요소:

* 핵심 업무 중단
* 고객 서비스 중단
* 매출 손실
* 대체업체 부재
* RTO 초과
* 법적 의무 수행 불가

### Security Risk

공급업체가 침해될 가능성과 조직에 미치는 보안 노출이다.

판단 요소:

* 취약한 보안통제
* 직접 Network Access
* 민감정보 처리
* 과거 침해
* 복잡한 Integration
* 다수의 Subprocessor

```text
Criticality = 실패했을 때 얼마나 아픈가?
Risk = 보안사고가 발생할 가능성과 노출이 얼마나 큰가?
```

공급업체는 다음과 같이 분류될 수 있다.

* Critical이면서 High-risk
* Critical이지만 Low-risk
* Non-critical이지만 High-risk
* Non-critical이고 Low-risk

두 축을 분리해야 평가 깊이와 통제 요구사항을 정확히 설정할 수 있다.

## 8. Waterfall에서 보안 결함 발견

Waterfall은 단계별 진행을 전제로 하지만 이전 단계로 절대 돌아갈 수 없다는 의미는 아니다.

중대한 Design Flaw 발견 시:

1. 결함과 위험 문서화
2. Change Request 제출
3. 영향·일정·비용 평가
4. 승인 후 Design 단계 수정
5. 관련 산출물 업데이트
6. 개발·테스트 재수행

임시 Workaround는 Root Cause를 제거하지 못한다. Critical Design Flaw를 Maintenance 단계까지 미루면 이후 수정 비용과 위험이 커진다.

> 결함이 발생한 계층에서 수정하는 것이 원칙이다.

## 9. Access Control vs Encryption

### RBAC

사용자의 역할에 따라 데이터 접근과 변경 권한을 제한한다.

* Confidentiality: 비인가 사용자의 접근 차단
* Integrity: 승인된 역할만 데이터 변경
* Availability: 인가된 사용자에게 필요한 접근 제공

### Encryption

키가 없는 주체가 내용을 읽지 못하도록 한다.

* 주 목적: Confidentiality
* 암호화만으로 업무 권한을 결정하지 못함
* 인가된 사용자의 오용을 차단하지 못함
* Availability를 직접 제공하지 않음

### Data Masking

민감 필드의 일부 또는 전체를 숨겨 노출을 줄인다.

### IDS

의심스러운 접근을 탐지하지만 접근 자체를 예방하지는 않는다.

질문의 문맥이 **Access Control Policy Review**라면 RBAC가 가장 직접적이다.

## 10. Access Review·Recertification

주기적인 Access Review의 보안 목적은 다음과 같다.

* 권한이 현재 직무와 일치하는지 확인
* 과도한 권한 제거
* Orphan Account 제거
* Dormant Account 제거
* SoD 위반 확인
* 규제 요구사항 증명

### 시험 관점

규제산업에서 “PRIMARY purpose”를 물으면 **Access Recertification 의무 준수**가 정답으로 제시될 수 있다.

### 실제 보안 관점

근본 목적은 **현재 권한이 승인되고 정당하며 최소 권한 원칙을 만족하는지 확인하는 것**이다. Compliance는 이를 요구하고 증명하게 만드는 외부 동인이다.

```text
Security Purpose = 적절한 권한 유지
Compliance Purpose = 주기적 검토와 증거 제출
```

## 11. Patch 우선순위

실제 Risk-based Patch Prioritization:

```text
Priority
= Vulnerability Severity
× Exploitability
× Asset Criticality
× Exposure
÷ Compensating Controls
```

주요 요소:

1. CVSS와 기술적 영향
2. Known Exploited Vulnerability 여부
3. Exploit Code 존재
4. Internet Exposure
5. 자산의 업무 중요도
6. 데이터 민감도
7. 보상 통제
8. Patch 적용 부작용

단순히 취약점이 오래됐다는 이유로 우선순위를 높이지 않는다. Vendor 순서도 조직의 실제 위험과 일치한다고 보장할 수 없다.

## 12. Patch 누락 발견: IR vs Process Improvement

Critical Patch가 적용되지 않은 상태를 발견하면 먼저 다음을 확인한다.

* 해당 취약점이 실제 악용됐는가
* 공격 흔적이 있는가
* 비정상 계정·Process·Network Connection이 있는가
* 침해 범위는 어디까지인가
* 증거를 보존해야 하는가

이후:

1. Containment
2. Patch 또는 Mitigation
3. Eradication
4. Recovery
5. Root Cause Analysis
6. Patch Process 개선

```text
Incident Response = 이미 침해됐는지 확인
Patch Management Improvement = 다시 발생하지 않도록 절차 개선
```

Patch부터 적용하면 악성 Process나 Persistence가 남을 수 있으며 일부 증거가 변경될 수 있다.

## 13. Data Owner와 Custodian

### Data Owner

* 데이터 분류 결정
* 접근 요구사항 결정
* 보존기간 결정
* 보호 수준 결정
* 위험 수용
* 최종 책임 보유

### Data Custodian

* 저장
* 백업
* 복구
* 접근통제 구현
* 파기 실행
* 시스템 운영

Cloud Provider가 Custodian 역할을 수행할 수는 있지만 조직의 Data Owner 책임은 사라지지 않는다.

```text
Operation can be outsourced.
Accountability cannot be outsourced.
```

계약에 포함할 내용:

* 역할과 책임
* 암호화
* 접근통제
* Incident Notification
* Backup·Recovery
* Data Location
* Data Return·Deletion
* Audit Right
* Subprocessor 제한

## 14. 암호 알고리즘 Decommission

오래된 알고리즘을 바로 삭제하면 기존 Ciphertext를 복호화할 수 없게 될 수 있다.

### 안전한 순서

1. 사용 위치 Inventory
2. 영향과 위험 평가
3. 저장 데이터·Protocol·Application 의존성 파악
4. 대체 알고리즘 선택
5. Key Migration 계획
6. 데이터 Re-encryption
7. 호환성 테스트
8. 기존 알고리즘 비활성화
9. 기존 Key 폐기
10. 완료 검증

```text
Assessment
→ Migration Plan
→ Re-encryption
→ Validation
→ Disable
→ Key Destruction
```

Re-encryption은 중요하지만 무엇을 다시 암호화해야 하는지 먼저 파악해야 한다.

## 15. Password Storage

비밀번호는 복호화할 필요가 없으므로 암호화가 아니라 **Salted Password Hashing**을 사용한다.

권장 구조:

```text
Password
+ Unique Salt
→ Argon2id / bcrypt / scrypt / PBKDF2
→ Password Hash 저장
```

피해야 할 방식:

* Cleartext
* MD5
* SHA-1
* Salt 없는 SHA-256
* Reversible Encryption
* 동일 Salt 재사용
* Source Code 내부 저장

추가 통제:

* Rate Limiting
* Account Lockout 또는 Progressive Delay
* MFA
* Breached Password Blocklist
* Secure Password Reset
* Session Protection

## 16. 다국적 데이터 규제

문항에서는 가장 엄격한 적용 규정을 공통 Baseline으로 삼는 방식을 정답으로 제시했다.

장점:

* 정책 일관성
* 관리 복잡도 감소
* 높은 공통 보호 수준
* 규제 누락 가능성 감소

그러나 실제로는 지역별로 충돌하거나 고유한 요구사항이 존재할 수 있다.

* Data Localization
* 보존기간
* 삭제 의무
* 국외 이전
* 동의 방식
* Employee Monitoring
* Breach Notification 기한

따라서 실무적으로는 다음 구조가 적절하다.

```text
Global Minimum Baseline
+ Regional Legal Addendum
+ Data-flow별 적용 법률 분석
```

“가장 엄격한 규제 하나를 전 세계에 무조건 적용”하는 것은 항상 가능한 해법은 아니다.

## 17. Secure Design Principles

### Zero Trust

* Never Trust, Always Verify
* 모든 요청 검증
* 지속적 인증·인가
* Device·User·Context 확인
* 최소 권한
* Micro-segmentation

### Least Privilege

사용자·서비스·Process에 필요한 최소 권한만 부여한다.

### Privacy by Design

개인정보 보호를 설계 초기부터 포함한다.

* Data Minimization
* Purpose Limitation
* Consent
* Retention
* Privacy Defaults

### Defense in Depth

서로 다른 통제를 여러 계층에 배치한다.

### Fail Secure

장애 발생 시 안전한 Deny 상태로 전환한다.

### Secure Defaults

초기·기본 설정이 안전해야 한다.

```text
Failure 발생 시 안전한가? = Fail Secure
초기 설정이 안전한가? = Secure Defaults
여러 통제를 겹치는가? = Defense in Depth
모든 요청을 검증하는가? = Zero Trust
권한 범위를 최소화하는가? = Least Privilege
```

---

## 18. NIST CSF vs ISO 27001 vs SABSA vs PCI DSS

### NIST Cybersecurity Framework

* Risk-based
* 유연한 적용
* 조직 규모·산업별 조정 가능
* 다양한 Framework·규제와 Mapping 가능
* Govern·Identify·Protect·Detect·Respond·Recover

### ISO/IEC 27001

* ISMS 요구사항
* Risk-based Management System
* 국제 인증 가능
* 정책·책임·지속적 개선 중점

### SABSA

* Business-driven Security Architecture Method
* 사업 요구와 Security Architecture 연결

### PCI DSS

* Payment Card Data 보호를 위한 산업 표준
* 적용 범위가 Cardholder Data Environment로 제한

```text
유연한 Cyber Risk Framework = NIST CSF
인증 가능한 ISMS = ISO 27001
Business-driven Architecture = SABSA
Payment Card Security = PCI DSS
```

NIST CSF와 ISO 27001은 모두 포괄적이다. “다양한 환경에 유연하게 적용”이라는 문맥에서는 NIST CSF가 선택될 수 있지만, 국제적인 인증과 경영시스템이 핵심이면 ISO 27001이 더 적합하다.

## 19. Authentication Baseline

문항은 Password + Complexity + Lockout을 최초 Baseline으로 제시했지만 그대로 일반화하면 안 된다.

현대적인 Customer Authentication:

* 긴 Passphrase 허용
* 알려진 유출 Password 차단
* Password Manager 허용
* 불필요한 Complexity Rule 최소화
* 강제 Periodic Rotation 지양
* MFA 또는 Passkey 제공
* Rate Limiting
* Risk-based Authentication
* Secure Recovery

### Hardware Token

강력하지만 모든 일반 고객에게 발급하면 비용과 운영 부담이 크다.

### Biometrics

단독 Server-side Credential이라기보다 사용자 Device에서 Passkey·MFA 잠금 해제 수단으로 사용하는 경우가 많다.

### Social Login

편리하지만 외부 Identity Provider에 대한 의존성이 생긴다.

> 가장 단순한 Password Login을 먼저 구현해야 한다는 문항 정답은 다소 낡은 관점이다. 새 시스템이라면 Phishing-resistant Passkey와 MFA를 초기 설계부터 고려하는 것이 더 적절하다.

```

## 21. 전력 통제

### UPS

* 정전 즉시 전력 공급
* 전압 변동·Surge 보호
* 장비의 무중단 운영
* Generator 기동 전까지 Bridge 역할

### Generator

* 장시간 정전 대응
* 시작까지 시간이 필요
* 연료와 정기 테스트 필요
* UPS와 함께 사용

한 장비 내부의 PSU 장애를 견디지만 건물 전원 전체가 끊기면 효과가 없다.

→ UPS가 즉시 공급
→ Generator가 기동
→ Generator가 장시간 공급
```

가장 좋은 구조는 Network·Server·Storage를 UPS에 연결하고 Generator와 서로 다른 Power Feed를 추가하는 것이다.

## 24. Fail Secure vs Defense in Depth

### Fail Secure

System Component가 실패할 때 권한을 허용하지 않고 안전하게 차단한다.

예:

* Authorization Service 장애 → Access Deny
* Door Controller 장애 → 위험에 따라 잠금 유지
* Firewall 오류 → Default Deny
* Transaction 검증 실패 → 거래 중단

### Defense in Depth

하나의 통제가 실패해도 다른 통제가 방어하도록 계층화한다.

예:

```text
MFA
+ Endpoint Security
+ Segmentation
+ Encryption
+ Monitoring
+ Backup
```

문제에서 **“when it fails”**라는 표현이 나오면 Fail Secure가 정답이다.

## 34. IPv6 Address Type

IPv6 주소 유형:

* **Unicast:** 한 Interface
* **Multicast:** 그룹의 여러 Interface
* **Anycast:** 여러 Interface 중 Routing상 가장 가까운 하나

IPv6에는 Broadcast 주소가 없다. IPv4 Broadcast 역할은 Multicast가 대신한다.

```text
IPv4 = Unicast + Multicast + Broadcast
IPv6 = Unicast + Multicast + Anycast
```

“Broadcast를 LEAST 사용해야 한다”기보다 **IPv6에서는 Broadcast 자체가 존재하지 않는다**가 정확하다.

## 36. 해설을 그대로 외우면 안 되는 항목

### Q3 — Log Review

문항은 Review Cadence를 정답으로 제시했다. “Review 활동”에 한정하면 타당하지만 전체 Monitoring Capability에서는 필요한 Event가 수집되는지 먼저 확인해야 한다.

### Q40 — RBAC와 CIA

RBAC가 CIA를 모두 완전히 보장하는 것은 아니다.

* RBAC → 접근과 변경 권한
* Encryption → 기밀성
* Backup·Redundancy → 가용성
* Hash·Signature → 무결성

문항이 Access Control Policy 문맥이므로 RBAC가 선택된 것이다.

### Q42 — Access Review 목적

보안의 근본 목적은 과도하거나 부적절한 권한을 제거하는 것이다. 규제 준수는 이를 주기적으로 수행하고 증명하게 만드는 동인이다.

### Q50 — Patch Severity

취약점 심각도만으로 Patch 순위를 정하면 안 된다. 실제로는 Asset Criticality, Exploitability, Exposure를 함께 고려해야 한다.

### Q63 — 가장 엄격한 규제

가장 엄격한 공통 Baseline은 유용하지만 국가별 Data Localization이나 삭제 의무가 충돌할 수 있다. Global Baseline + Regional Requirement가 더 현실적이다.

### Q65 — Zero Trust

Zero Trust는 광범위한 Access Risk를 줄이지만 SQL Injection, Business Logic Flaw, Memory Corruption 같은 모든 애플리케이션 공격을 직접 방어하지는 않는다.

실제 안전한 Architecture:

```text
Zero Trust
+ Least Privilege
+ Secure Coding
+ Defense in Depth
+ Input Validation
+ Encryption
```

### Q68 — NIST CSF vs ISO 27001

NIST CSF만 Holistic한 것은 아니다. ISO 27001도 조직 전체를 대상으로 하는 Risk-based ISMS다.

* Adaptability·Mapping → NIST CSF
* 국제 인증·Management System → ISO 27001

### Q76 — Password First

Password Complexity와 Lockout을 최초 해답으로 보는 것은 낡은 문항이다. 신규 고객 Portal이라면 Passkey·MFA·Breached Password 차단을 초기 설계부터 고려하는 편이 더 안전하다.

### Q86 — FedRAMP

FedRAMP는 미국 Federal Cloud에 특화되어 있다. 일반 민간기업이나 한국 조직의 CSP 평가에 무조건 가장 적합한 것은 아니다.

### Q87 — SBOM

SBOM은 Component Visibility를 제공할 뿐 Software Integrity를 단독으로 보장하지 않는다. Code Signing, SCA, Provenance, Vendor Governance가 함께 필요하다.

### Q93 — Data Center 내부 통제

Server MFA와 Physical Access Control은 서로 대체 관계가 아니다.

* 방에 누가 들어가는가 → Physical Access
* 서버에 누가 로그인하는가 → MFA·PAM

### Q147 — Threat Intelligence Diversity

많은 Source보다 관련성·정확성·적시성이 더 중요할 수 있다. Diversity는 Blind Spot을 줄이는 보조 기준으로 이해한다.

## 37. 문제 풀이 기준

### 질문의 범위를 유지한다

* Network Segmentation → Network Control
* Internal Audit → 조직이 직접 통제하는 영역
* Access Control Policy → RBAC·Authorization
* Failure Behavior → Fail Secure
* Cloud Provider Assurance → Cloud 관련 Assurance
* Component Vulnerability → SBOM·SCA

### 선행 단계와 후속 단계를 구분한다

```text
Assess → Plan → Implement → Validate → Improve
```

* 암호 알고리즘 폐기 → 영향 평가가 먼저
* Patch 누락 → 침해 여부 확인이 먼저
* 공급업체 관리 → 위험과 역할 확인 후 계약
* Design Flaw → Change Control 후 설계 수정

### 근본 원인을 해결한다

* VPN 우회 → VPN 성능 개선
* Server Room이 공용공간에 있음 → 안전한 구역으로 이전
* Third-party Component 불명 → SBOM으로 가시성 확보
* Standing Privilege → JIT로 제거

### 통제의 작동 시점을 확인한다

* UPS → 정전 즉시
* Generator → 장시간
* Detection → 사고 확인
* Prevention → 사고 발생 가능성 감소
* Recovery → 사고 후 복구
