# CISSP 오답노트 5

## 1. 바로 외울 것

### 보안 평가·테스트

* **BAS:** 알려진 공격 기법을 지속적·자동으로 실행하여 기존 통제의 탐지·차단 능력 검증
* **Vulnerability Assessment:** 취약점을 넓게 식별
* **Penetration Test:** 취약점을 실제 악용하여 영향 입증
* **Red Team:** 목표 중심의 수동·비정기적 Adversary Emulation
* **Purple Team:** 공격팀과 방어팀이 협력하여 탐지·대응 개선
* **Backup 검증:** 실제 Restore Test
* **Backup Verification KRI:** 복구 테스트 실패율 또는 요구 시간 초과율
* **Penetration Test Report:** 모든 Finding을 뒷받침하는 정확한 기술적 증거가 기반
* **Synthetic Test만 실패:** 직전 Application/API 변경이 Script를 깨뜨렸는지 먼저 확인

### IAM·접근통제

* **Dormant Account 비율:** Account Management의 대표 KRI
* **PEP:** PDP가 내린 접근 결정을 실제 요청에 적용
* **PDP:** 정책을 평가하여 Allow/Deny 결정
* **SSO 핵심 위험:** 하나의 Credential 침해가 모든 연동 Application으로 확산
* **Access Review 핵심:** 검토자가 사용자의 실제 업무를 이해해야 함
* **ABAC:** 사용자·자원·행위·환경 속성을 요청 시점에 평가
* **RBAC:** 직무 역할별로 권한을 묶어 할당
* **TACACS+:** 전체 Payload 암호화, AAA 분리, 명령어 단위 권한·기록
* **RADIUS:** 주로 Network Access 인증
* **NAC + Legacy:** 엄격한 기본정책에 승인된 예외를 관리

### 네트워크·클라우드

* **VPC 핵심 설계:** Workload를 Subnet으로 분리하고 Routing·Security Policy 통제
* **Public Wi-Fi:** 내부망과 Firewall로 분리
* **Micro-segmentation:** Network Topology와 무관하게 Workload별 Policy 적용
* **Out-of-band Management:** Production Network와 관리 경로를 분리
* **Out-of-band 핵심 이점:** Production Network 위협과 장애로부터 관리 Plane 격리
* **WPA3:** SAE, Offline Dictionary Attack 저항, Forward Secrecy, PMF
* **WPA2:** AES-CCMP지만 WPA3보다 약함
* **IPv6:** IPsec 지원이 Protocol Suite에 포함되지만 자동 보안은 아님
* **API 보호:** Application-layer AuthN/AuthZ·입력검증·Rate Limit이 핵심
* **API Gateway:** 인증·Rate Limit·Routing·Logging 등 공통 API 통제 중앙화

### 운영·복구

* **EOS Application:** 교체 전까지 Segmentation·Virtual Patching·접근제한·모니터링 등 보상 통제
* **DR Team Activation 첫 단계:** Communication Plan으로 담당자 소집
* **DR Lessons Learned:** Communication이 실제로 작동했는지 평가
* **BC/DR Plan 핵심 데이터:** 핵심 인력·외부업체의 최신 연락처
* **IR Procedure 핵심:** 역할·책임·권한·Escalation 경로 명확화
* **HVAC 기본 목적:** 온도와 습도를 허용 범위에서 안정적으로 유지
* **Dual Power Supply:** 장비 내부 Power Supply 장애 제거
* **UPS:** 순간 정전 대응
* **Generator:** 장기 정전 대응

### 소프트웨어·변경관리

* **Version Control 보호의 기반:** Repository에 대한 엄격한 접근통제
* **Signed Commit:** 변경 작성자와 Commit 무결성 검증
* **DevSecOps Change Management:** 배포 전 보안 요구사항 충족 여부 검증
* **Application Allowlisting:** 승인된 실행파일만 허용하여 Malware 실행 예방
* **Open-source 장기 안전성:** CVE 처리 절차와 실제 Patch 제공 이력 확인
* **대규모 IT·OT 변경:** 단계적 배포로 영향과 장애 범위 제한

### 데이터·거버넌스

* **Cloud Migration FIRST:** Cloud-specific Risk Assessment
* **Supply Chain FIRST:** Provider가 충족해야 할 최소 보안 요구사항 정의
* **BYOD 보안·프라이버시 균형:** Work Container 분리
* **BYOD 분실 대응:** Full-disk Encryption
* **암호키 관리:** 위험 기반 Cryptoperiod와 자동 Rotation
* **Data Classification:** Proprietary Design·Client Data는 일반적으로 Confidential
* **Security Mission Alignment:** 개별 통제보다 핵심 업무가 중단되지 않도록 보안 의사결정 기준 설정
* **조직의 윤리 의지:** 시행·위반처리 절차가 포함된 Code of Conduct
* **외부 감사의 구조적 장점:** 경영진으로부터의 독립성과 객관성

## 2. BAS vs VA vs Penetration Test vs Red/Purple Team

| 활동                       | 목적                        | 방식         | 빈도      |
| ------------------------ | ------------------------- | ---------- | ------- |
| Vulnerability Assessment | 취약점 식별                    | 주로 자동화·광범위 | 정기·빈번   |
| Penetration Test         | 실제 악용 가능성과 영향 입증          | 수동 중심      | 정기·비정기  |
| BAS                      | 현재 통제가 공격 기법을 탐지·차단하는지 검증 | 자동화        | 지속적     |
| Red Team                 | 목표 달성을 위한 현실적 공격 시뮬레이션    | 수동·은밀      | 비정기     |
| Purple Team              | 공격·방어팀 협업으로 탐지 개선         | 수동·협업      | 프로젝트 단위 |

```text
VA = 어떤 약점이 존재하는가?
Pentest = 그 약점으로 무엇을 할 수 있는가?
BAS = 현재 통제가 공격을 탐지·차단하는가?
Red Team = 실제 공격자처럼 목표를 달성할 수 있는가?
Purple Team = 공격 결과를 이용해 방어를 어떻게 개선할까?
```

**Continuous + Automated + Control Validation**이 동시에 나오면 BAS다.

## 4. Software Configuration Management

Version Control System은 배포되는 Software의 Source of Truth다.

### 통제 우선순위

```text
누가 접근할 수 있는가
→ 누가 변경했는가
→ 무엇을 변경했는가
→ 변경이 안전한가
→ 모든 활동이 기록됐는가
```

| 통제                  | 역할                     |
| ------------------- | ---------------------- |
| Repository RBAC     | 비인가 접근·변경 예방           |
| Protected Branch    | 직접 Push·무단 Merge 차단    |
| Pull Request Review | 변경 검토                  |
| Signed Commit       | 작성자·Commit 무결성 검증      |
| Secret Scanning     | Hardcoded Secret 탐지    |
| SAST·SCA            | Code·Dependency 취약점 탐지 |
| Audit Logging       | 변경 추적                  |

Signed Commit도 중요하지만 Repository 접근권한이 과도하면 승인된 사용자가 악의적 변경을 수행할 수 있다.

## 6. 전력 가용성

| 통제              | 방어하는 장애               |
| --------------- | --------------------- |
| Dual PSU        | 장비 내부 Power Supply 장애 |
| Dual Power Feed | 한 전력 Feed·Circuit 장애  |
| UPS             | 순간 정전·전압 변동           |
| Generator       | 장시간 정전                |
| Redundant Site  | 시설 전체 장애              |

이상적인 구조:

```text
Device A PSU → Power Path A → UPS A
Device B PSU → Power Path B → UPS B
                         ↓
                    Generator
```

Dual PSU가 있어도 두 PSU가 같은 PDU나 같은 Circuit에 연결되면 실질적 Redundancy가 아니다.

## 7. Backup 검증

### 통제별 목적

* **Backup Frequency:** RPO 개선
* **Encryption:** 기밀성
* **Checksum:** File Integrity
* **Restore Test:** 실제 복구 가능성
* **Geographic Copy:** Site 장애 대응
* **Immutable Backup:** Ransomware·악성 삭제 대응

실제 Restore Test는 다음을 검증한다.

* Backup Media가 읽히는가
* 데이터가 완전한가
* 암호키를 사용할 수 있는가
* Application이 정상 시작되는가
* 의존 시스템이 존재하는가
* 실제 RTO를 충족하는가
* 담당자가 절차를 수행할 수 있는가

```text
Backup Success ≠ Recovery Success
```

### Backup Verification KRI

가장 직접적인 위험지표:

* Restore Test 실패율
* Restore 시간 초과율
* 손상된 Backup 비율
* 검증되지 않은 Backup 비율
* Recovery Point 누락률

암호화된 Backup 비율은 보호 상태를 나타내지만 **복구 가능성의 KRI**는 아니다.

## 11. Account Management KRI

### 위험을 직접 보여주는 지표

* Dormant Account 비율
* Terminated User Account의 비활성화 지연시간
* Orphan Account 수
* 과도한 Privilege 보유자 수
* Access Review 미완료율
* Shared Account 수
* MFA 미적용 Privileged Account 수

### 단순 운영지표

* 전체 Account 증가량
* 성공 Login 횟수
* Password Reset 횟수

### Lockout Event

Lockout 증가는 다음을 의미할 수 있다.

* Brute-force Attack
* Credential Stuffing
* Password Usability 문제
* 잘못된 Service Credential

유용한 탐지지표지만 Account Lifecycle 위험을 가장 직접적으로 나타내는 것은 Dormant·Orphan Account다.

## 13. Synthetic Transaction 장애 분석

실제 사용자는 정상인데 Synthetic Script만 실패하고, 직전에 Application Update가 있었다면 다음을 확인한다.

* Authentication Endpoint 변경
* Parameter·Header 변경
* Token Flow 변경
* CSRF Token 추가
* Redirect 변경
* MFA·CAPTCHA 추가
* API Version 변경
* Response Schema 변경

오류별 단서:

| 오류                   | 가능성                                |
| -------------------- | ---------------------------------- |
| Authentication Error | Credential·API Contract·Token Flow |
| TLS Handshake Error  | Certificate·Protocol·Cipher        |
| HTTP 403/WAF Block   | WAF·Bot Detection·IP Policy        |
| Timeout              | Network·Performance·Dependency     |
| Selector Not Found   | UI 변경                              |

```text
가장 최근 변경
+ 오류 유형
+ 영향을 받는 대상
= 가장 가능성 높은 원인
```

## 15. HVAC

서버실 HVAC의 기본 목적은 장비가 요구하는 환경을 안정적으로 유지하는 것이다.

### 환경 위험

* 고온 → 부품 수명 단축·과열
* 저습도 → 정전기
* 고습도 → 응결·부식
* 급격한 변화 → Thermal Cycling
* 먼지 → 냉각 방해·전기적 문제

### 설계 요소

* 적정 온도·습도
* Hot/Cold Aisle
* Airflow
* Sensor
* Redundant Unit
* Dust Filtration
* 독립 전원
* Preventive Maintenance

```text
Primary Outcome = 안정적인 환경
Redundancy = 그 환경을 장애 중에도 유지하는 수단
```

## 16. Wireless Security

| --------------- | ---------------------------------------- |
| WEP             | RC4·Static Key, 폐기                       |

## 17. BYOD

### Containerization

Work Profile·Managed Container 안에 기업 Application과 Data를 분리한다.

장점:

* 기업 Data만 암호화·정책 적용
* 기업 영역만 Remote Wipe
* Personal App이 기업 Data에 접근하지 못함
* 조직이 개인 사진·메시지에 접근하지 않음

### Full-device MDM

보안 통제력은 강하지만 개인 Device 전체를 관리하여 Privacy 침해 가능성이 있다.

### Endpoint Baseline

* Full-disk Encryption
* Screen Lock
* 최신 Patch
* Root/Jailbreak 차단
* EDR·Mobile Threat Defense
* Work Container

## 20. Cryptographic Key Management

### Key Lifecycle

```text
Generation
→ Distribution
→ Storage
→ Use
→ Rotation
→ Revocation
→ Archive·Recovery
→ Destruction
```

### Key Rotation 목적

* 노출 기간 제한
* 한 Key로 보호되는 데이터량 제한
* 알고리즘·정책 변경 반영
* 직원·시스템 변경 대응
* Compromise 영향 제한

Rotation 주기는 다음에 따라 결정한다.

* Key 종류
* 데이터 민감도
* 사용량
* 위협
* Algorithm Strength
* Regulatory Requirement

무조건 짧은 주기로 바꾸는 것이 아니라 적절한 **Cryptoperiod**를 정한다.

### 위험한 구조

하나의 장기 Master Secret이 침해되면 모든 Derived Key가 영향을 받을 수 있다. Master Key도 HSM·KMS로 보호하고 Rotation·Versioning해야 한다.

## 36. Application Allowlisting vs EDR

### Application Allowlisting

* 승인된 실행파일만 허용

* 예방 통제

### EDR

* Process·Behavior Monitoring
* 탐지
* 조사
* 격리

* Credential Theft

따라서 실제 환경에서는 Allowlisting + EDR을 함께 사용한다.

## 37. 해설을 그대로 외우면 안 되는 항목

### M&A Due Diligence

Security Policy Alignment만으로 인수 위험을 충분히 평가할 수 없다. 실제로는 Regulatory Liability, 과거 Breach, Technical Debt, 개인정보 처리, 계약 의무가 거래가치에 직접 영향을 줄 수 있다.

### Penetration Test Report

Technical Detail은 기반이지만 독자가 경영진이라면 Business Impact와 Remediation Priority가 핵심일 수 있다. “누구를 위한 보고서인가”를 함께 본다.

### Dual Power Supply

Dual PSU가 별도 Power Path에 연결되지 않으면 실질적 Redundancy가 아니다. Facility 전체 가용성에는 Dual Utility Feed·UPS·Generator도 필요하다.

### WPA3

SAE는 WPA3-Personal의 특징이다. Corporate Environment에서는 WPA3-Enterprise와 EAP-TLS가 더 적절할 수 있다.

### Key Rotation

무조건 정기적으로 자주 Rotation하는 것이 최선은 아니다. Key Type과 위험에 따른 Cryptoperiod를 사용해야 한다.

### External Audit

외부감사의 독립성이 더 높더라도 결과가 자동으로 더 정확한 것은 아니다. 상반된 Finding은 Scope·Evidence·Sampling·Criteria를 비교하여 조정해야 한다.

### Online Voting과 Private Key

투표 내용 자체를 개인의 Private Key와 직접 결합하면 Ballot Secrecy와 익명성을 훼손할 수 있다.

안전한 전자투표는 다음을 함께 만족해야 한다.

* Voter Eligibility
* 한 사람당 한 표
* Ballot Secrecy
* Vote Integrity
* Verifiability
* Coercion Resistance
* Non-repudiation 또는 감사 가능성

```text
유권자 인증
≠ 투표 내용과 신원 영구 결합
```

문항의 “Ballot을 Voter Private Key에 Binding”은 일반적인 전자투표 설계 원칙으로 그대로 외우면 안 된다.

### GDPR와 DPO

모든 조직이 DPO를 반드시 임명해야 하는 것은 아니다. 일반적으로 먼저 해야 할 일은 다음과 같다.

1. GDPR 적용 여부 확인
2. Personal Data Inventory와 Data Flow Mapping
3. Controller·Processor 역할 확인
4. Lawful Basis 확인
5. Gap Assessment
6. 필요한 경우 DPO 임명

대규모 정기 Monitoring이나 특별범주 데이터의 대규모 처리가 있을 때 DPO가 요구될 수 있다.

### PII Handling과 DPO

DPO는 독립적으로 준수 여부를 감독하는 역할이며 직접 모든 Handling Requirement를 작성·운영하는 소유자가 아니다. 요구사항은 법무·Privacy·Data Owner·Security가 함께 정의해야 한다.

### IPv6 IPsec

IPv6가 IPsec을 지원한다고 해서 모든 IPv6 Traffic이 자동으로 암호화·인증되는 것은 아니다. IPsec을 실제로 구성하고 사용하는지가 중요하다.

### BYOD Full-disk Encryption

Full-disk Encryption은 Device가 꺼진 상태에서 분실됐을 때 효과적이다. Unlock된 Device, Malware, Data Leakage에는 Containerization·Conditional Access·EDR 등이 필요하다.

### 대규모 IT·OT 변경

Phased Rollout은 위험을 줄이지만 Coordination의 핵심은 통합 Change Plan, 공동 책임자, Communication, Rollback이다. 단계적 배포만으로 부서 간 조정이 자동 해결되지는 않는다.

### API Gateway vs Input Validation

API Gateway는 공통 보안정책을 중앙화하지만 Application 내부의 Injection·Business Logic Flaw를 모두 방지하지 않는다.

```text
Gateway = Traffic Management·Common Policy
Input Validation = 악성 입력 방어
Authorization = 객체·기능 접근 방어
```

질문의 위협 범위를 확인해야 한다.
