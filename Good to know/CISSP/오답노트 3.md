# CISSP 오답노트 3

## 1. 바로 외울 것

### IAM·특권 접근

* **Privileged Access 위험 감소:** 특권 사용자를 최소한의 검증된 인원으로 제한
* **PAM Session Recording:** 오용 탐지·추적을 위한 Detective Control
* **JIT Access:** 필요할 때만 짧은 시간 특권을 부여하고 자동 만료
* **JIT의 1차 효과:** Standing Privilege 제거
* **JIT의 2차 효과:** 정기 Access Certification 대상과 업무량 감소
* **Cloud Account 최우선 보호:** MFA
* **Adaptive/Risk-based Access:** 위험 점수에 따라 Step-up Authentication 수행
* **ABAC:** 사용자·자원·행위·환경 속성을 정책과 비교
* **Policy Exception:** 위험 평가 후 경영진의 공식 Risk Acceptance 필요

### 가용성·복구

* **최고 수준 Fault Tolerance:** Active-Active Data Center
* **Load Balancing/Cluster:** 서버 장애 대응
* **RAID 1:** 디스크 장애 대응
* **Backup:** 장애를 견디는 것이 아니라 장애 후 복구
* **Parallel DR Test:** 복구 환경을 실제 가동하되 운영 시스템은 계속 유지
* **Full Interruption:** 운영을 실제 중단하고 복구 환경으로 전환
* **Backup 검증:** 실제 Restore Test가 가장 확실
* **Patch Management 첫 단계:** 시스템·소프트웨어·버전 Inventory 구축

### 네트워크

* **SDN 핵심 위험:** 중앙 Controller와 Northbound/Southbound API
* **Micro-segmentation:** Hypervisor-embedded Firewall·Host-based Firewall
* **VLAN:** Broadcast Domain 분리이지 세밀한 Workload 통제는 아님
* **Third-party Support:** 지원 대상 장비로만 범위가 제한된 VPN
* **Split Tunneling:** 인터넷 트래픽이 조직의 검사·로깅을 우회하므로 보안상 불리
* **물리적 전송매체 보호:** Armored Fiber·Conduit
* **Microservices 통신:** 서비스 간 TLS 또는 mTLS
* **IoT 핵심 보호:** 장치와 관리 시스템 간 통신의 기밀성·무결성
* **Network Device Hardening:** 불필요한 서비스·포트 비활성화

### AAA 프로토콜

| 프로토콜         | 핵심 용도·특징                                                |
| ------------ | ------------------------------------------------------- |
| **RADIUS**   | 유·무선 네트워크 접근 인증, 기존 Directory 연동                        |
| **TACACS+**  | 네트워크 관리자 접속, AAA 분리, 명령어 단위 권한통제                        |
| **Diameter** | 차세대 AAA, Peer-to-Peer, Capability Negotiation, Failover |

### 소프트웨어 보안

* **API 무단 접근 방지:** 모든 Endpoint에 Authentication + Authorization
* **Input Validation:** Injection·비정상 입력 방지
* **Code Signing:** 실행코드의 출처와 무결성 검증
* **Hardcoded Secret:** 코드에서 제거하고 외부 설정 또는 Secret Vault에 저장
* **CI/CD 첫 보안검사:** SAST + SCA
* **Buffer Overflow Exploit Mitigation:** Stack Canary + ASLR + DEP/NX + FORTIFY_SOURCE
* **SAST/Fuzzing:** 취약점 발견
* **Compiler/OS Mitigation:** 남은 취약점의 악용을 어렵게 함
* **민감 데이터를 처리하는 기능:** 필요한 데이터만 접근하도록 Least Privilege 적용
* **STRIDE:** 위협 식별·분류
* **DREAD:** 식별된 위협의 우선순위 평가

### 데이터·암호화

* **End-to-End Encryption:** 중간 서비스 제공자까지 포함해 콘텐츠를 읽거나 변경할 수 없게 보호
* **Provider-managed Encryption:** 제공자가 키를 관리하므로 제공자에 대한 신뢰 필요
* **Silicon Root of Trust:** 하드웨어 정품성과 Firmware 무결성을 암호학적으로 검증
* **TPM:** 개별 시스템 내부의 키·측정값 보호
* **HSM:** 중앙화된 고가치 키 생성·보관·암호 연산
* **Code Repository Integration:** 저장·전송 구간 암호화
* **데이터 폐기:** 보존·파기 관련 법률과 규제 요구사항을 먼저 확인

### 물리 보안

* **Remote Branch:** 출입구의 물리적 Access Control이 기본
* **Server Room:** 출입 인증과 접근통제가 핵심
* **Campus 출입:** 카드 기반 전자 출입통제는 보안과 편의성의 균형
* **고등급 Backup Media:** Fire-rated Safe 등 환경·화재 보호 필요

## 2. 가장 중요한 문제 풀이 원칙

### Prevention > Detection

위험을 직접 줄이는 예방 통제가 단순 탐지·감사 통제보다 우선한다.

| 목적           | 더 직접적인 통제               | 보조 통제             |
| ------------ | ----------------------- | ----------------- |
| 특권계정 위험 감소   | 특권계정 수 최소화              | 세션 녹화             |
| API 무단 접근 방지 | AuthN/AuthZ             | Logging           |
| Phishing 감소  | Email Gateway Filtering | 훈련·시뮬레이션          |
| 원격계정 탈취 방지   | MFA                     | Password Training |
| 버퍼오버플로 악용 방지 | Canary·ASLR·DEP         | SAST·Fuzzing      |

단, 질문이 **탐지·조사·증거 확보**를 묻는다면 Monitoring과 Logging이 정답이 된다.

### Elimination > Hardening > Monitoring

불필요한 위험 요소를 제거하는 것이 남겨둔 뒤 방어하는 것보다 강하다.

```text
불필요한 특권 제거
→ 남은 계정 강화
→ 활동 모니터링
→ 사고 발생 후 조사
```

### FIRST와 PRIMARY는 단계가 중요

* **FIRST:** 현재 상황에서 가장 먼저 해야 할 단계
* **PRIMARY:** 다른 통제를 지배하는 근본 목적·기준
* **MOST EFFECTIVE:** 목표를 가장 직접적으로 달성
* **BEST:** 위험·비용·운영·법률을 종합한 선택
* **LEAST:** 가장 부적절하거나 가장 큰 보안 공백을 만드는 선택

## 3. JIT·PAM

### Privileged Account 최소화

특권계정이 많을수록 다음 위험이 증가한다.

* 탈취할 Credential 증가
* Insider Threat 증가
* 불필요한 Standing Privilege 증가
* Access Review 범위 증가
* 사고 조사 범위 증가

따라서 특권 접근 위험을 가장 직접적으로 줄이는 방법은 **특권 보유 인원과 계정을 최소화하는 것**이다.

### JIT Access

```text
요청 → 승인 → 일시적 권한 상승 → 자동 만료
```

주요 효과:

1. Standing Privilege 제거
2. 권한 남용 시간 축소
3. 권한 상승 기록 생성
4. 자동 만료
5. 정기 Access Certification 업무량 감소

Access Certification은 지속적으로 존재하는 권한을 재검토하는 절차다. JIT는 지속 권한 자체를 줄이므로 검토 대상도 감소한다.

## 4. Fault Tolerance 범위

| 통제                      | 견딜 수 있는 장애            |
| ----------------------- | --------------------- |
| RAID                    | 디스크 장애                |
| Cluster·Load Balancer   | 서버·애플리케이션 장애          |
| Redundant Network/Power | 특정 인프라 장애             |
| Active-Passive Site     | 주 사이트 장애 후 전환         |
| Active-Active Site      | 사이트 전체 장애에도 즉시 서비스 유지 |
| Backup                  | 장애를 견디지 못하고 사후 복구     |

> Fault Tolerance는 장애 중에도 계속 운영하는 능력이고, Backup은 장애 후 데이터를 복구하는 수단이다.

## 5. DR 테스트

| 테스트                   | 실제 복구 시스템 가동 | 운영 중단 |
| --------------------- | -----------: | ----: |
| Read-through/Tabletop |            X |     X |
| Walkthrough           |            X |     X |
| Simulation            |        일부 절차 |     X |
| Parallel              |            O |     X |
| Full Interruption     |            O |     O |

### 문제 판별

* 계획과 역할을 토론 → **Tabletop**
* 절차를 단계별 검토 → **Walkthrough**
* 복구 환경을 실제 가동하되 고객 서비스 유지 → **Parallel**
* 실제 운영을 중단하고 전환 → **Full Interruption**

새로운 핵심 고객 시스템의 복구 능력을 운영 중단 없이 검증하려면 Parallel Test가 적절하다.

## 6. API 보안

### 위협별 직접 통제

| 위협               | 직접적인 통제                              |
| ---------------- | ------------------------------------ |
| 무단 접근            | Authentication + Authorization       |
| 다른 사용자의 객체 접근    | Object-level Authorization           |
| Injection        | Input Validation·Parameterized Query |
| Credential Abuse | MFA·Rate Limiting                    |
| 정보노출             | 최소 응답·암호화                            |
| 부인방지·조사          | Logging                              |
| 서비스 남용           | Rate Limit·Quota                     |

API Endpoint 경로를 숨기는 것은 Access Control이 아니다.

```text
Authentication = 누구인가?
Authorization = 이 사용자가 이 객체에 이 행위를 할 수 있는가?
```

## 7. Secret Management

### Hardcoded API Key 문제

코드에 Secret을 넣으면 다음 위치에 모두 복제된다.

* Git Repository
* Developer Laptop
* Build Artifact
* CI/CD Log
* Container Image
* Backup
* Fork와 Clone

### 안전한 순서

1. Secret을 Source Code에서 제거
2. Secret Manager·Vault 또는 외부 설정에 저장
3. 파일·Vault 접근권한 최소화
4. Runtime에 필요한 시점에만 주입
5. 주기적 Rotation
6. Repository Secret Scanning 적용
7. 노출된 기존 Key 즉시 폐기

문항 보기에서 Vault가 없다면 Webroot 밖의 제한된 설정 파일이 차선책이다.

> Obfuscation은 Secret을 숨길 뿐 제거하지 않는다.

## 8. SDN 보안

SDN은 Control Plane을 중앙 Controller로 집중시킨다.

* **Northbound API:** Controller ↔ Application·Orchestrator
* **Southbound API:** Controller ↔ Switch·Router

Controller가 침해되면 공격자는 다음을 변경할 수 있다.

* Flow Rule
* Routing
* Segmentation
* ACL
* Traffic Mirroring
* Network Visibility

따라서 핵심 통제는 다음과 같다.

* 강력한 Controller Authentication
* API Authorization
* TLS·mTLS
* Controller Hardening
* Role Separation
* API Logging
* Controller Redundancy

> SDN의 장점인 중앙집중화가 동시에 가장 큰 위험이다.

## 9. Programming Paradigm과 Side Channel

Side Channel 위험은 프로그래밍 패러다임 이름 자체보다 다음 특성에서 증가한다.

* Shared Mutable State
* Global State
* Secret-dependent Branch
* Secret-dependent Memory Access
* 동적 Lookup
* Cache와 실행시간 변화

상대적 위험:

```text
Immutable + Pure Function
< Encapsulated Mutable State
< Global Shared Mutable State
```

암호 구현의 핵심은 특정 언어 패러다임보다 **Constant-time Implementation**이다.

* Secret에 따른 분기 금지
* Secret에 따른 메모리 주소 접근 금지
* 일정한 연산량 유지
* 검증된 암호 라이브러리 사용

## 11. Egress Monitoring

Anomaly Detection은 정상 상태를 모르면 불가능하다.

먼저 파악할 것:

* 정상 Destination
* 정상 Protocol
* 정상 전송량
* 정상 시간대
* 사용자·서버별 업무 패턴
* 정기적인 대용량 전송
* 승인된 Cloud Service

```text
Baseline → Deviation Detection → Alert → Investigation
```

Real-time Alerting은 중요하지만 Baseline이 없으면 오탐이 폭증하거나 실제 유출을 놓친다.

## 12. Risk Tolerance와 비용

중복 전력·Active-Active·보안 통제의 투자 수준은 조직이 허용할 수 있는 위험에 의해 결정된다.

* **Risk Appetite:** 전체적인 위험 수용 성향
* **Risk Tolerance:** 특정 서비스에서 허용 가능한 구체적 한도
* **ROI:** 비용과 기대효과를 계산하는 입력
* **Compliance:** 반드시 지켜야 하는 최소 제약

예:

```text
서비스 중단 허용한도 5분
→ 현재 예상 중단시간 4시간
→ 허용 범위 초과
→ 추가 이중화 필요
```

ROI는 판단 자료지만 최종 결정 기준은 Risk Tolerance와 사업 요구다.

## 13. Third-party Remote Access

안전한 제3자 원격지원은 다음 조건을 만족해야 한다.

* 지원 대상 장비로만 Network Scope 제한
* MFA
* 개별 사용자 계정
* JIT·Time-bound Access
* 세션 승인
* Session Recording
* 접속 종료 후 권한 회수
* 활동 Logging
* 계약상 보안 의무

### 보기 비교

* **Scoped VPN:** 암호화 + 네트워크 수준 Least Privilege
* **Restricted Account:** 신원과 권한만 제한하며 네트워크 범위는 넓을 수 있음
* **Leased Line:** 전용 경로지만 자체적으로 인증·암호화·권한통제를 제공하지 않음
* **Vendor Tool:** 공급업체 경로와 통제 수준을 추가로 신뢰해야 함

## 14. Risk-based Access vs ABAC

### ABAC

정책이 속성을 평가한다.

```text
Subject + Resource + Action + Environment → Allow/Deny
```

예:

* 직원 부서
* 데이터 분류
* 접속 시간
* 장치 상태
* 접속 위치

### Risk-based Adaptive Access

현재 이벤트의 위험 점수를 계산하여 인증 강도를 조절한다.

```text
정상 로그인 → 일반 인증
새 국가 + 새 장치 → MFA
고액 송금 + 이상 행동 → 추가 인증·차단
```

* **속성이 정책을 만족하는가?** → ABAC
* **현재 행동이 얼마나 위험한가?** → Risk-based Access

## 15. Split Tunneling

### Full Tunnel

모든 트래픽이 조직의 VPN Gateway를 통과한다.

장점:

* 중앙 검사
* URL Filtering
* DLP
* Malware Inspection
* 일관된 Logging

단점:

* 대역폭과 지연 증가

### Split Tunnel

기업 대상 트래픽만 VPN을 통과하고 인터넷 트래픽은 로컬로 직접 나간다.

위험:

* 인터넷 트래픽 검사 우회
* 감염된 Endpoint가 기업망과 인터넷 사이 Bridge 역할
* 중앙 Logging·DLP 우회
* 정책 일관성 감소

고보안 환경에서는 Full Tunnel이 일반적으로 더 적절하다.

## 16. Buffer Overflow 통제

### 취약점 발견 통제

* SAST
* Code Review
* Fuzzing
* DAST
* Sanitizer

### Exploitation Mitigation

* **Stack Canary:** Return Address 변조 전 Stack 손상 탐지
* **ASLR:** 주소 예측 방해
* **DEP/NX:** 데이터 영역에서 코드 실행 차단
* **FORTIFY_SOURCE:** 위험 함수 호출 강화
* **PIE:** 실행파일 주소 Randomization 지원
* **Hardened Allocator:** Heap 손상 탐지

질문이 “버그를 찾는다”면 SAST/Fuzzing, “남은 버그의 악용을 막는다”면 Compiler·OS Mitigation이 우선이다.

## 17. 시스템 폐기

### CISO 수준의 우선순위

1. 법적 보존기간 확인
2. Litigation Hold 확인
3. 규제상 파기 요구 확인
4. 승인된 Sanitization 기준 결정
5. 파기 증거와 Chain of Custody 유지

### 기술 실행

* **Clear:** 일반적인 논리적 삭제·덮어쓰기
* **Purge:** 복구가 매우 어렵도록 강한 제거
* **Destroy:** 물리적 파괴

CISO의 PRIMARY concern은 “어떤 도구를 쓸 것인가”보다 **언제 무엇을 어떻게 파기해야 법적 요구를 만족하는가**이다.

---

## 18. MFA vs Account Lifecycle

### MFA

현재 활성 계정의 Credential 탈취를 방어한다.

* Phishing
* Password Reuse
* Credential Stuffing
* Password Database 유출

### Provisioning·Deprovisioning

계정 생명주기를 관리한다.

* 입사
* 직무 변경
* 퇴사
* 휴직
* 권한 회수
* Orphan Account 제거

인터넷에 노출된 Cloud 서비스의 즉각적인 계정 탈취 위험에는 MFA가 직접적인 통제다. Lifecycle Automation은 이를 보완한다.

## 19. STRIDE vs DREAD

### STRIDE

| 항목 | 위협                     |
| -- | ---------------------- |
| S  | Spoofing               |
| T  | Tampering              |
| R  | Repudiation            |
| I  | Information Disclosure |
| D  | Denial of Service      |
| E  | Elevation of Privilege |

### DREAD

* Damage
* Reproducibility
* Exploitability
* Affected Users
* Discoverability

### 순서

```text
Architecture 분석
→ STRIDE로 위협 식별
→ DREAD 등으로 우선순위 결정
→ 통제 설계
→ 검증
```

초기 설계 단계에서는 먼저 위협을 찾아야 하므로 STRIDE가 우선이다.

## 20. EOS 장비

End of Support 이후에는 새로운 취약점에 대한 보안 패치가 중단될 수 있다.

유지 가능 조건:

* 공식 Extended Support 계약
* 위험평가 완료
* 경영진 Risk Acceptance
* 네트워크 격리
* 기능 제한
* 강화된 Monitoring
* 교체 일정과 예산 확정

부적절한 정당화:

```text
비용 절감 효과는 계산했지만 보안 위험은 계산하지 않음
```

정량화된 비용과 Unquantified Risk를 비교해 “비용이 더 중요하다”고 판단할 수는 없다.

## 21. Encryption Scope

| 방식                          | 보호 범위                   |
| --------------------------- | ----------------------- |
| Link Encryption             | 개별 통신 링크                |
| TLS                         | Client와 TLS Endpoint 사이 |
| Site-to-Site VPN            | VPN Gateway 사이          |
| Provider-managed Encryption | 제공자가 복호화 가능한 구조         |
| End-to-End Encryption       | 송신자부터 최종 수신자까지          |

글로벌 음성·영상·파일 협업에서 중간 제공자를 신뢰하지 않고 기밀성과 무결성을 확보하려면 E2EE가 가장 강하다.

단, 다음 요구가 있으면 E2EE와 충돌할 수 있다.

* Server-side DLP
* Compliance Recording
* Content Inspection
* eDiscovery

## 22. RADIUS vs TACACS+ vs Diameter

### RADIUS

* 사용자 네트워크 접근
* Wired·Wireless·VPN
* 중앙 Directory 연동
* UDP 1812/1813
* Authentication과 Authorization이 밀접하게 결합

### TACACS+

* Router·Switch 관리자 접속
* TCP 49
* 전체 Payload 암호화
* Authentication·Authorization·Accounting 분리
* Command-level Authorization

### Diameter

* 차세대 통신·이동통신 AAA
* TCP 또는 SCTP
* Peer-to-Peer
* Capability Negotiation
* Failover와 확장성

```text
Network Access = RADIUS
Device Administration = TACACS+
Next-generation Telecom AAA = Diameter
```

## 23. Responsible Disclosure

윤리적인 공개 절차:

1. 유지관리자에게 비공개로 취약점과 Patch 제공
2. 수신 확인
3. 합리적인 수정 기간 협의
4. 필요 시 CERT/CC 등 조정기관 활용
5. Patch 제공 후 조정된 공개

직접 유지관리자와 안전하게 협력할 수 있다면 먼저 비공개 통보하는 것이 일반적이다. 공급업체가 응답하지 않거나 다수 조직 조정이 필요하면 CERT가 적절하다.

## 24. Policy Exception

1. 업무상 필요성 문서화
2. 위험평가
3. 보상 통제 결정

* 책임과 배상
* 데이터 반환·삭제
* 규제 준수

```text
Due Diligence → Risk Decision → Contract → Monitoring
```

계약 조건 검토보다 공급업체 자체의 보안평가가 먼저다.

## 29. TPM vs HSM

### TPM

* 한 Endpoint나 Server 내부
* Device-bound Key
* Secure Boot·Measured Boot 지원
* Disk Encryption Key 보호
* Platform Attestation

### HSM

* 중앙 서버·서비스
* 대규모 Key Management
* CA Private Key
* Database Master Key
* Signing·Encryption Operation
* 높은 물리적 변조 방지

문항의 HSM 선택지가 “인증서 발급 전용”으로 제한되고, 시스템 자체의 키 관리가 필요하다면 TPM이 더 적합할 수 있다.

## 30. Code Signing

Code Signing은 다음을 검증한다.

* Publisher Identity
* 코드가 서명 후 변경되지 않았는지
* 승인된 배포 Artifact인지

Access Control은 누가 파일을 변경할 수 있는지 제한하지만 파일 자체의 출처와 무결성을 수신자에게 증명하지는 못한다.

```text
Write Permission = 변경 예방
Code Signing = 출처·무결성 검증
```

## 39. SOC 2 Type II Evidence Gap

SOC 2 Type II는 일정 기간 통제가 실제로 지속 운영되었는지 평가한다.

증거가 부족할 때:

* 존재하지 않는 과거 증거를 만들어내지 않음
* 통제를 즉시 강화
* 앞으로 완전한 증거를 수집
* 과거 Gap을 감사인에게 투명하게 공개
* 필요하면 감사기간·범위 조정 협의
* 보상 통제 증거가 실제 존재하면 제공

## 42. 문항 해설을 그대로 외우면 안 되는 항목

### Code Repository의 Encryption

문항은 Encryption을 MOST critical로 제시했지만 실제 Repository 보안에서는 다음도 동등하게 중요하다.

* Repository·Branch별 RBAC
* MFA
* Signed Commit
* Branch Protection
* Secret Scanning
* CI/CD Token 보호

“Integration 중 이동·저장되는 코드의 기밀성”에 초점을 둘 때 Encryption을 선택한다.

### Hardcoded API Key의 외부 설정 파일

Webroot 밖의 설정 파일은 코드 내부 저장보다 낫지만 최선은 아니다.

실제 권장:

```text
Secret Manager/Vault
> 권한 제한된 외부 설정
> 코드 내부 암호화
> Obfuscation
```

### Programming Paradigm Side Channel

Prototype-based Programming 자체가 항상 가장 위험한 것은 아니다. Side Channel은 실제 구현, Runtime, Compiler Optimization, Cache Access와 Constant-time Discipline에 좌우된다.

문항에서는 **Implicit + Shared + Mutable + Global State**가 가장 많은 선택지를 고른 것이다.

### Sensitive Data 기능의 Least Privilege

문항은 Least Privilege를 정답으로 제시했지만 Encryption과 Input Validation도 필수다.

* 데이터 접근 범위를 줄임 → Least Privilege
* 저장·전송 중 노출 방지 → Encryption
* 악성 입력 방지 → Input Validation

“오류 발생 시 피해 범위 제한”에 초점을 둘 때 Least Privilege가 우선이다.

### IoT Transmission Security

모든 IoT 환경에서 통신 보안이 항상 최우선인 것은 아니다.

* 배치 후 운영 통신 → Transmission Security
* 외부에 노출된 장비 → Physical Tampering
* 구매 단계 → Supply Chain
* Cloud 연동이 핵심 → Third-party Integration

### TPM과 HSM

일반적으로 의료정보 시스템의 중앙 Key Management에는 HSM이 더 강할 수 있다. 이 문항은 HSM 선택지를 인증서 발급 전용으로 제한했기 때문에 TPM을 정답으로 삼은 것으로 보인다.

### Defense in Depth vs Encryption

“민감 데이터 보호”라는 표현만으로는 다소 모호하다.

* 데이터 기밀성의 직접 통제 → Encryption
* 전체 시스템 위험 감소 → Defense in Depth

### Server Room의 Biometric Access

Biometric이 항상 카드보다 우수한 것은 아니다. 비용, 오인식, 장애 대응, 프라이버시, Tailgating 통제를 함께 고려해야 한다.

문항에서는 서버실 출입통제가 벽 강화보다 직접적이라는 의미로 이해한다.

### Backup Media의 Fire-rated Safe

화재 보호는 중요하지만 최고등급 매체에는 다음도 함께 필요하다.

* Dual Control
* 출입 Logging
* 온도·습도 통제
* Off-site Storage
* 암호화
* Inventory
* Chain of Custody

```text
4. 기술적 통제인가, 경영 판단인가?
5. 특정 기술이 새롭게 만드는 고유 위험은 무엇인가?
6. 보기 중 목표를 가장 직접적으로 달성하는 것은 무엇인가?
7. 한 단계 앞이나 뒤의 보기를 선택하고 있지 않은가?
8. 범위가 가장 넓거나 피해를 근본적으로 제거하는 통제는 무엇인가?
```

### 반복 실수 방지

* 탐지보다 위험 제거를 먼저 고려
* “좋은 통제”가 아니라 “질문의 위협을 직접 막는 통제” 선택
* 기술적 성능과 사업 전략 구분
* FIRST 문제에서는 선행 단계 확인
* LEAST 문제에서는 보기의 장점보다 포기하는 보안 기능 확인
* CISO 관점에서는 기술 실행보다 법률·위험·경영 승인 우선
